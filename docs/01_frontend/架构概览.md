# Frontend 架构概览

## 1. 当前技术基线

- `React`
- `TypeScript`
- `Vite`
- `react-router-dom`

## 2. 当前目录

```text
frontend/src/
├─ App.tsx
├─ main.tsx
├─ router.tsx
├─ index.css
├─ layouts/
├─ components/
├─ lib/
└─ pages/
```

## 3. 页面边界

- `/`：HomePage，全局状态入口，已实现。
- `/discover`：搜索批次与候选论文。
- `/acquire`：待获取、待补 PDF、待人工确认队列。
- `/library`：正式文献库列表、筛选和状态统计。
- `/library/:paperId`：Paper Detail，单篇论文详情与产物检查。
- `/config`：数据根、路径健康、parser 配置。

当前代码中除 HomePage 外仍以占位路由为主。后续补页面时，应优先按上面的业务边界调整路由，不再保留与 Paper 全流程无关的 `runtime / logs` 独立入口，除非它们被收敛进 Config 或 Paper Detail。

## 4. API 对接

前端通过 `frontend/src/lib/api.ts` 调用后端接口，当前已对接：

- `GET /api/dashboard/home`

下一步需要补齐：

- `GET /api/dashboard/discover`
- `GET /api/dashboard/acquire`
- `GET /api/dashboard/library`
- `GET /api/papers`
- `GET /api/papers/{paper_id}`
- `POST /api/papers/{paper_id}/parse-pdf`
- `GET /api/config`

默认 API 根地址：

- `http://127.0.0.1:8000`
- 可通过 `VITE_API_BASE_URL` 覆盖

## 5. UI 原则

- 视觉风格参考 `docs/04_reference/frontend/screenshots/HomePage.png`
- 当前不引入完整设计系统
- 页面优先表达流程状态、人工待办和失败原因
- 复杂编辑先放到 Paper Detail 和 Config，不在列表页堆叠表单

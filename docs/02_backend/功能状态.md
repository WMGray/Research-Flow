# 后端功能状态与文档缺口

## 1. 文档目标

本文档记录 Research-Flow 后端“当前代码已经实现什么、哪些只是目标设计、哪些文档存在缺口”。它用于连接 `backend/app`、`backend/core`、`backend/tests` 与 `docs/02_backend`、`docs/03_api`，避免架构设计文档被误读为已经全部落地。

进一步的逐功能风险、可能故障表现和修复顺序见 [全局风险排查.md](./全局风险排查.md)。

核查时间：2026-04-26

核查依据：

- `backend/app/main.py`
- `backend/app/api/*.py`
- `backend/app/schemas/*.py`
- `backend/core/schema.py`
- `backend/core/services/papers/*`
- `backend/core/services/projects.py`
- `backend/core/services/paper_download/service.py`
- `backend/tests/test_papers_api.py`
- `backend/tests/test_projects_api.py`
- `backend/tests/test_paper_download_api.py`
- `backend/tests/test_pdf_parser_pipeline.py`

## 2. 状态标记

| 标记 | 含义 |
|------|------|
| 已实现 | 已有路由、schema、service 或测试覆盖，可按当前行为使用 |
| 部分实现 | 有入口或局部能力，但还不是目标设计中的完整闭环 |
| 设计中 | 文档中有设计，代码尚未提供可用入口 |
| 文档缺口 | 当前文档与代码行为不一致，或缺少实现状态说明 |

| Paper artifact/run 审计 | 已实现 | 新增 `biz_paper_artifact`、`biz_paper_pipeline_run`，并提供 `/artifacts` 与 `/pipeline-runs` 查询入口 |

| `note.md` 生成 | 部分实现 | 可从 canonical sections 生成带 `RF:BLOCK_START/END` 的 note；尚未实现增量合并、锁定片段和冲突处理 |
| Knowledge 抽取 | 部分实现 | 当前写出 `extracted/knowledge.json` 空占位结果；没有 `biz_knowledge` 表和 Knowledge CRUD API |
| Dataset 抽取 | 部分实现 | 当前写出 `extracted/datasets.json` 空占位结果；没有 `biz_dataset` 表和 Dataset CRUD API |
| Project 资源管理 | 部分实现 | 支持创建、列表、详情、更新；自动初始化六个 Markdown 文档；尚未提供删除接口 |
| Project 文档读写 | 已实现 | 支持六模块文档 `overview`、`related_work`、`method`、`experiment`、`conclusion`、`manuscript` 的读写与版本冲突检测 |
| Project 论文关联 | 已实现 | 支持关联、解除关联、列出关联论文；关系写入 `asset_link.relation_type` |
| Project 自动生成任务 | 设计中 | related work / method / experiment / conclusion / manuscript 生成接口尚未实现 |
| Job 查询与取消 | 部分实现 | 支持列表、详情、取消非终态 job；当前动作多为同步执行后写入终态 job，不是 Celery 异步队列 |
| Category | 设计中 | 文档列为 P0，但代码没有 category route 与 `biz_category` 表 |
| Config / Agent / Prompt API | 设计中 | 配置文件与 refine skill 绑定已存在；`/api/v1/config/*`、`sys_agent_profile`、`sys_prompt_template`、`sys_skill_binding` 尚未实现 |
| Presentation / Feed / Conference / Graph | 设计中 | 当前没有后端 API 和业务表 |
| Zotero MCP | 部分实现 | `/services/mcp` 与 `/services/mcp/tools` 暴露脱敏配置与工具列表；尚未接入 `/api/v1/zotero/*` 导入链路 |

## 6. 已确认文档缺口

| 缺口 | 影响 | 当前处理 |
|------|------|----------|
| API 文档把目标接口与已实现接口混写 | 前端或后端后续实现可能误以为 Category、Config、P1 资源已可用 | 本文档列出已挂载接口，并在 `接口规范.md` 增加当前实现说明 |
| 通用 envelope 文档与实现不一致 | 实现返回 `data/meta/error`，错误多为 FastAPI `detail`；文档仍写 `code/message/data/request_id` | 在 `接口规范.md` 标注当前 envelope 与目标契约差异 |
| 表设计文档没有说明当前 schema 是精简子集 | 可能按目标字段写查询导致运行期失败 | 在 `表设计.md` 增加当前实现对照 |
| Paper 下载与解析文档描述为完整链路 | 当前 Paper download/parse 主要是本地占位动作，真实 gPaper/MinerU 尚未接入 PaperService | 在本文档和模块文档标注“部分实现” |
| Knowledge / Dataset 文档描述为落库资源 | 当前只写空占位 JSON，没有业务表和 CRUD | 在本文档标注为部分实现 |
| Project 文档列出删除与模块生成任务 | 当前没有删除路由，也没有 Project 级生成任务 API | 在本文档和 API 文档标注缺口 |
| Config / Agent / Prompt 被列为 P0 API | 当前只有文件配置和 refine runtime，没有 `/api/v1/config/*` | 在本文档标注为设计中 |
| docs 目录索引缺少功能状态页 | 后续无法快速判断文档可信层级 | 已新增本文档并更新 `docs/README.md` |

## 7. 后续优先修正文档事项

1. 将 `接口规范.md` 分为“当前实现 API”和“目标 API”两个层级，避免同一表格混用。
2. 为 `Paper`、`Project`、`Job` 拆出 `docs/03_api/endpoints/*.md`，每个接口给出当前 request / response 示例。
3. 按 [PDF 解析与 LLM 优化设计](./services/pdf-parser.md) 细化 `paper_refine_parse` 的 diagnose / repair / verify request 和 response 示例。
4. 选择一个方向统一 job 表命名：继续使用 `jobs`，或迁移为 `biz_job` 并同步测试。
5. 明确 `/paper-download/*` 是否保留为 legacy route，或迁移到 `/api/v1/papers/{paper_id}/download`。
6. 在 `表设计.md` 中为“目标字段”和“当前字段”增加版本标签。
7. 在路线图中明确 Category、Config API、Project delete、Knowledge/Dataset CRUD 的真实落地顺序。

# Research-Flow 表设计文档

## 1. 整体架构与设计理念

我们采用统一的**“三层架构”**作为整个系统的基础分级结构。系统中的所有表，无论是核心业务数据、长文本内容，还是底层二进制文件，**全部归入这三层之中**。这种设计实现了真正的“数据资产与业务逻辑解耦”。

### 1.1 三层定义全局规范

- **物理层 (Physical Layer)**：只管存**真实的、重负荷的数据内容**（如 OSS 里的二进制文件流、大段的 Markdown 文本）。该层彻底剥离业务属性，只关心数据的纯粹存储、哈希去重和安全机制。
- **资产层 (Asset Layer)**：只管存**实体本身的属性与逻辑身份**（如论文的标题/作者、文件的 ID/名字、文档的归属属性）。这一层定义了系统中客观存在的所有“资产”。
- **引用层 (Reference Layer)**：只存**动态的业务关联与关系**（如“课题 A”挂载了“文件 X”、“论文 B”引用了“数据集 C”）。

### 1.2 核心机制

- **引用映射代替物理拷贝**：同一份数据（不论是一份 PDF 还是某段共享的长文本），物理层永远只存 1 份，通过引用层建立多条关联记录进行复用。
- **安全回收（引用计数）**：物理层与资产层引入“引用计数”。业务层删除关系仅仅是断开引用层记录并递减计数，只有当计数彻底归零（所有业务都不再使用该底层数据）时，才触发真实的物理文件删除和垃圾回收。

## 2. 物理层（Physical Layer）

专职负责庞大、非结构化或半结构化数据的真实物理存储。它们是被引用的“死物”，自身不带任何业务状态。

### 2.1 二进制文件内容 (`file_blob`)
只管存真实的二进制数据（如 OSS 里的文件流），关心文件存哪了、哈希值是多少，负责底层级别的物理去重。
- `blob_id` (主键)
- `content_sha256` (文件哈希，避免同一篇 PDF 在磁盘存两份)
- `content_length` (文件大小)
- `storage_path` (S3/OSS/本地的物理存储路径)
- `ref_count` (引用计数，记录有多少个 asset 指向它，归零时触发物理文件删除)
- `created_at`

### 2.2 长文本内容记录 (`doc_version`)
负责真实的大段 Markdown 长文本或 JSON 富文本存储，实现修改留痕和多版本共存。
- `id` (主键)
- `doc_id` (关联资产层 `doc_entry.id`)
- `content` (Markdown 长文本内容)
- `generator_type` (生成者类型：`human` 用户手写, `llm` 大模型自动生成)
- `prompt_template_id` (如果是 LLM 生成，记录所用的 Prompt 模板 ID)
- `created_at` (真实内容产生时间)

---

## 3. 资产层（Asset Layer）

负责定义系统中的各类实体“资产”。这些资产有自己的身份和轻量级的结构化属性，但不包含沉重的物理正文，也不包含复杂的网状关联。

### 3.1 业务核心资产

**论文资产 (`paper`)**
- `id` (主键)
- `title` (论文标题)
- `authors` (作者列表)
- `publication_year` (发表年份)
- `venue` (来源会议/期刊)
- `abstract_text` (摘要纯文本)
- `status` (`in_library`, `daily_push`, `ignored`)
- `zotero_item_id` (外部标识 Zotero ID)

**课题资产 (`project`)**
- `id` (主键)
- `name` (课题名称)
- `description` (简短描述)
- `status` (`active` 进行中, `archived` 已归档)

**数据集资产 (`dataset`)**
- `id` (主键)
- `name` (数据集名称)
- `description` (描述)
- `access_url` (获取途径)

**结构化知识资产 (`knowledge_item`)**
- `id` (主键)
- `type` (知识点分类：
  - 观点类（可引用的论断）：`trend_evaluation` 领域现状评述, `method_critique` 现有方法批判, `core_claim` 核心主张, `key_conclusion` 关键结论
  - 定义与特征类（客观知识）：`task_definition` 任务定义, `feature_definition` 特征定义, `feature_extraction` 特征提取)
- `title` (知识点标题)
- `summary_zh` (中文简短概括/定义解释/公式流程描述)
- `original_en` (英文原文依据，观点类必填，定义与特征类可选/留空)
- `source_paper_id` (最初提取来源的论文 ID)

**会议资产 (`conference`)**
- `id` (主键)
- `name` (会议名称)
- `abstract_deadline` / `paper_deadline` / `notification_date`
- `timezone`

### 3.2 逻辑衍生资产

**文件逻辑资产 (`file_asset`)**
只管存文件本身的逻辑属性（即“这是一个什么文件”）。
- `asset_id` (主键)
- `blob_id` (关联物理层 `file_blob`)
- `file_name` (原始文件名)
- `file_ext` (如 `pdf`, `docx`, `png`)
- `file_md5` (文件级校验值)
- `source_type` (`manual`, `zotero`, `export`)
- `ref_count` (记录有多少个引用指向它，归零时可清理该逻辑资产)

**文档逻辑资产 (`doc_entry`)**
定义一篇长文档的从属户口与业务类型。
- `id` (主键)
- `target_type` (归属实体类型：`paper`, `project`)
- `target_id` (归属实体 ID)
- `doc_type` (`auto_analysis`, `manual_note`, `project_overview` 等)
- `current_version_id` (指向物理层 `doc_version` 的当前最新版本内容 ID)

## 4. 引用层（Reference Layer）

专职处理一切多对多的“挂载”与“关联”关系，解耦资产间的直接绑定。

### 4.1 业务文件挂载关系 (`file_ref`)
只存业务对象与文件资产之间的挂载关系（如“课题 A”关联了“实验结果 X”）。业务层增删文件，仅仅是操作这张表的记录和递减引用计数。
- `ref_id` (主键)
- `asset_id` (关联资产层 `file_asset`)
- `biz_domain` (业务域，如 `paper`, `project_experiment`)
- `biz_id` (业务实体 ID，如具体的论文 ID)
- `ref_role` (引用角色，如 `source_pdf` 原文PDF, `attachment` 实验附件)

### 4.2 业务资产图谱关联 (`kg_edge`)
表达核心业务实体资产之间一切多对多的网状关系。
- `id` (主键)
- `source_type` (起始节点类型，如 `paper`, `knowledge_item`)
- `source_id` (起始节点 ID)
- `target_type` (目标节点类型，如 `dataset`, `paper`, `project`)
- `target_id` (目标节点 ID)
- `relation_type` (关系类型语义：
  - `paper_uses_dataset`: 论文使用数据集
  - `paper_cites_paper`: 论文引用论文
  - `paper_belongs_project`: 论文关联课题
  - `viewpoint_supports_paper`: 观点支持论文)
- `metadata` (关系附带属性，如在此次使用中的评估指标、实验数值)
- `created_type` (`auto` 系统抽取, `manual` 人工绑定)

## 5. 核心流转示例：在一套三层架构中运作

**场景：用户将一篇包含 PDF 的文献入库，系统自动生成了摘要分析，并提取了其使用的数据集。**

1. **物理层就绪（安放重物）**：
   - PDF 文件流存入 OSS，计算哈希存入 `file_blob`。
   - 大模型生成的 Markdown 摘要长文本，存入 `doc_version`。
2. **资产层落位（登记户口）**：
   - 记录文献本身的基础元数据入 `paper` 表。
   - 记录 PDF 的名称、格式入 `file_asset` 表。
   - 建立摘要分析文档的实体记录入 `doc_entry` 表。
   - 如果大模型提取到的数据集是全新的，则在 `dataset` 表新建记录。
3. **引用层连线（构建网状关系）**：
   - 将文献与 PDF 文件联系起来 -> 在 `file_ref` 中写入挂载记录，并将底层文件引用计数 +1。
   - 将文献与数据集的“使用关系”联系起来 -> 在 `kg_edge` 中写入 `paper_uses_dataset` 关系。
   - （`doc_entry` 的文本关联由于是一对多版本追踪，通过 `current_version_id` 和 `doc_id` 自行闭环映射）。。

# Research-Flow 表设计文档

## 1. 架构原则：极致解耦的三层模型

本系统遵循“物理-资产-引用”三层架构原则，实现底层存储与业务逻辑的彻底分离。

- **物理层 (Physical Layer)**：**“生存层”**。只记录物理实体的唯一 ID、存储引擎和访问路径。它不关心文件叫什么，只关心文件“在不在”以及“怎么拿”。
- **资产层 (Asset Layer)**：**“本体层”**。记录实体的客观属性（如 MD5、大小、文件名、逻辑类型）。它是对物理实体的第一层业务解释，是系统内所有逻辑引用的主体。
- **引用层 (Reference Layer)**：**“连线层”**。记录资产之间多对多的拓扑关系。它不存储资产属性，仅表达资产间的逻辑网状结构。

---

## 2. 第一层：物理层 (Physical Layer)
物理层表设计保持极致精简，仅作为存储介质的指针。

### 2.1 物理存根表 (`physical_item`)
| 字段名 | 类型 | 约束 | 描述说明 |
| :--- | :--- | :--- | :--- |
| `item_id` | `UUID` | 主键 | 物理实体的唯一 ID |
| `storage_path` | `VARCHAR(512)` | 非空 | **Payload：** 磁盘真实路径或 URI |
| `storage_type` | `VARCHAR(20)` | 默认 'LOCAL' | 存储引擎（LOCAL, S3, OSS, MINIO） |
| `ref_count` | `INT` | 默认 0 | **引用计数**。被资产层引用的次数，归零时物理删除 |

---

## 3. 第二层：资产层 (Asset Layer)
资产层是系统中的“数字资产登记簿”，包含了所有与存储位置无关的客观描述性元数据。

### 3.1 资产登记表 (`asset_registry`)
| 字段名 | 类型 | 约束 | 描述说明 |
| :--- | :--- | :--- | :--- |
| `asset_id` | `BIGINT` | 主键, 自增 | 全局资产唯一逻辑 ID |
| `item_id` | `UUID` | 外键 | **关联物理层**。指向 `physical_item.item_id` |
| `display_name` | `VARCHAR(255)` | 非空 | 资产的逻辑显示名称（如 "Abstract.md"） |
| `file_ext` | `VARCHAR(20)` | 索引 | 物理文件扩展名（pdf, md, png 等） |
| `file_size` | `BIGINT` | - | 文件大小（字节），用于校验与展示 |
| `content_hash` | `VARCHAR(64)` | 唯一索引 | **内容指纹**（MD5 或 SHA-256），用于资产级查重 |
| `asset_type` | `VARCHAR(30)` | 索引 | **业务大类**：Paper, Project, Markdown, Dataset, Image, Dir |
| `is_deleted` | `BOOLEAN` | 默认 False | 逻辑删除标记 |
| `created_at` | `TIMESTAMP` | 自动生成 | 资产登记时间 |
| `updated_at` | `TIMESTAMP` | 自动更新 | 资产属性最后变更时间 |

---

## 4. 第三层：引用层 (Reference Layer)
引用层只负责建立 M:N 的逻辑连线，不关心资产的具体属性。

### 4.1 资产关系连线表 (`asset_link`)
| 字段名 | 类型 | 约束 | 描述说明 |
| :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | 主键, 自增 | 关系记录 ID |
| `source_id` | `BIGINT` | 索引 | 起始资产 ID（关联 `asset_registry.asset_id`） |
| `target_id` | `BIGINT` | 索引 | 目标资产 ID（关联 `asset_registry.asset_id`） |
| `relation_type` | `VARCHAR(50)` | 索引 | **关系语义**：`CONTAINS` (包含), `REFERENCES` (引用), `DERIVED_FROM` (衍生) |

---

## 5. 业务逻辑层 (Business Logic Layer)

业务表不直接参与存储管理，而是通过 `asset_id` 挂载到资产层。这一层负责具体科研业务逻辑的实现。

### 5.1 核心实体模块

#### 1. 文献业务表 (`biz_paper`)
存储入库文献的学术元数据。
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `asset_id` | `BIGINT` | **主键**。关联 `asset_registry.asset_id` |
| `title` | `VARCHAR(512)` | 论文名字 |
| `authors` | `JSON` | 论文作者列表 |
| `pub_year` | `INT` | 年份 |
| `venue` | `VARCHAR(255)` | 会议/期刊全称 |
| `venue_short` | `VARCHAR(50)` | 会议/期刊缩写 |
| `doi` | `VARCHAR(100)` | DOI |
| `zotero_id` | `VARCHAR(100)` | 关联zotero_id（后续与zotero联动） |
| `status` | `VARCHAR(20)` | 论文状态 |
| `category_id` | `BIGINT` | 分类ID，关联 `biz_category.id` |

#### 2. 课题业务表 (`biz_project`)
管理研究课题的生命周期状态。
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `asset_id` | `BIGINT` | **主键**。关联 `asset_registry.asset_id` |
| `name` | `VARCHAR(255)` | 课题名称 |
| `status` | `VARCHAR(20)` | `active` (进行中), `archived` (已归档) |
| `description` | `TEXT` | 课题的目标与核心研究问题简述 |

#### 3. 数据集业务表 (`biz_dataset`)
独立管理的研究数据集档案。
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `asset_id` | `BIGINT` | **主键**。关联 `asset_registry.asset_id` |
| `name` | `VARCHAR(255)` | 数据集名字 |
| `task_type` | `VARCHAR(100)` | 任务类别 |
| `description` | `TEXT` | 数据集描述 |
| `access_url` | `VARCHAR(512)` | 数据集链接 |

#### 4. 观点提取表 (`biz_view`)
存储从文献中抽取的观点（Views）。
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `asset_id` | `BIGINT` | **主键**。关联 `asset_registry.asset_id` |
| `view_type` | `VARCHAR(50)` | 观点类别 |
| `summary_zh` | `TEXT` | 观点中文 |
| `original_en` | `TEXT` | 观点英文原文 |
| `research_field` | `VARCHAR(100)` | 观点对应的领域 |

#### 5. 定义提取表 (`biz_definition`)
存储从文献中抽取的定义（Definition）。
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `asset_id` | `BIGINT` | **主键**。关联 `asset_registry.asset_id` |
| `def_type` | `VARCHAR(50)` | 定义类别 |
| `summary_zh` | `TEXT` | 定义中文总结 |
| `original_en` | `TEXT` | 英文原文 |
| `research_field` | `VARCHAR(100)` | 对应的领域 |

### 5.2 文档组织模块

#### 6. 文档布局映射表 (`biz_doc_layout`)
负责将 Paper 和 Project 的各个细分章节组织起来。
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `map_id` | `BIGINT` | 主键 |
| `parent_id` | `BIGINT` | Project/Paper ID（所属主体的资产 ID） |
| `doc_id` | `BIGINT` | doc_id（具体的 Markdown 资产 ID） |
| `doc_name` | `VARCHAR(255)` | doc名字 |
| `doc_path` | `VARCHAR(512)` | doc路径 |
| `doc_role` | `VARCHAR(50)` | doc角色（如 Abstract, Method 等） |

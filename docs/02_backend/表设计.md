# Research-Flow 表设计文档

## 1. 设计目标

后端表设计需要同时满足以下要求：

- 支撑本地文件与数据库索引协同管理
- 支撑 `Paper`、`Project`、`Presentation`、`Dataset`、`Knowledge` 五类核心资源
- 支撑 `note.md` 单笔记策略
- 支撑“PDF 解析 -> LLM 优化解析 -> 章节划分 -> note 生成”的处理链路
- 支撑异步任务、模型配置与 SkillPackage 管理
- 为 `P1` 的 Daily Feed、`P2` 的 Recommendation / `Zotero` / 会议追踪预留扩展空间

---

## 2. 分层原则

本系统采用“物理层 - 资产层 - 关系层 - 业务层”的四段式设计。

### 2.1 物理层

物理层只关心文件真实位置与存储引擎。

### 2.2 资产层

资产层为所有文件、目录与逻辑对象提供统一的全局标识。

### 2.3 关系层

关系层记录资源之间的显式关联，承担图结构与引用网络的基础功能。

### 2.4 业务层

业务层保存各类科研资源的领域字段，并通过 `asset_id` 挂接到资产层。

| 字段名 | 类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `artifact_id` | `BIGINT` | 主键，自增 | 产物记录 ID |
| `paper_id` | `BIGINT` | 外键 | 关联 `biz_paper.asset_id` |

| `artifact_key` | `VARCHAR(100)` | 联合唯一 | 稳定产物 key，如 `source_pdf`、`raw_markdown` |

### 4.12 Skill Catalog（文件系统，不建表）

Paper skill 以 `skills/{skill-name}/` 为 source of truth，不设计独立模型指令配置表。

约定：

- `SKILL.md` 保存能力边界、流程和验证方法。
- `agents/openai.yaml` 保存 UI 展示元数据。
- `references/runtime-instructions.md` 保存默认 runtime instructions。
- 需要拆分的 runtime instructions 使用同一个 reference 文件中的 `<!-- stage:... -->` 标记。
- 少量特殊入口可以放在 `references/{name}-runtime-instructions.md`，由 `skill_runtime.py` 按约定读取。

未来如果需要线上版本化，应管理 skill 包版本或 skill release manifest，而不是把模型指令复制到独立模板表。

# Research-Flow 表设计文档

## 1. 架构原则：极致解耦的三层模型

本系统遵循“物理-资产-引用”三层架构原则，实现底层存储与业务逻辑的彻底分离。

- **物理层 (Physical Layer)**：**“生存层”**。只记录物理实体的唯一 ID、存储引擎和访问路径。它不关心文件叫什么，只关心文件“在不在”以及“怎么拿”。
- **资产层 (Asset Layer)**：**“本体层”**。记录实体的客观属性（如 MD5、大小、文件名、逻辑类型）。它是对物理实体的第一层业务解释，是系统内所有逻辑引用的主体。
- **引用层 (Reference Layer)**：**“连线层”**。记录资产之间多对多的拓扑关系。它不存储资产属性，仅表达资产间的逻辑网状结构。

---

## 2. 第一层：物理层 (Physical Layer)
物理层表设计保持极致精简，仅作为存储介质的指针。

### 2.1 物理存根表 (`physical_item`)
| 字段名 | 类型 | 约束 | 描述说明 |
| :--- | :--- | :--- | :--- |
| `item_id` | `UUID` | 主键 | 物理实体的唯一 ID |
| `storage_path` | `VARCHAR(512)` | 非空 | **Payload：** 磁盘真实路径或 URI |
| `storage_type` | `VARCHAR(20)` | 默认 'LOCAL' | 存储引擎（LOCAL, S3, OSS, MINIO） |
| `ref_count` | `INT` | 默认 0 | **引用计数**。被资产层引用的次数，归零时物理删除 |

---

## 3. 第二层：资产层 (Asset Layer)
资产层是系统中的“数字资产登记簿”，包含了所有与存储位置无关的客观描述性元数据。

### 3.1 资产登记表 (`asset_registry`)
| 字段名 | 类型 | 约束 | 描述说明 |
| :--- | :--- | :--- | :--- |
| `asset_id` | `BIGINT` | 主键, 自增 | 全局资产唯一逻辑 ID |
| `item_id` | `UUID` | 外键 | **关联物理层**。指向 `physical_item.item_id` |
| `display_name` | `VARCHAR(255)` | 非空 | 资产的逻辑显示名称（如 "Abstract.md"） |
| `file_ext` | `VARCHAR(20)` | 索引 | 物理文件扩展名（pdf, md, png 等） |
| `file_size` | `BIGINT` | - | 文件大小（字节），用于校验与展示 |
| `content_hash` | `VARCHAR(64)` | 唯一索引 | **内容指纹**（MD5 或 SHA-256），用于资产级查重 |
| `asset_type` | `VARCHAR(30)` | 索引 | **业务大类**：Paper, Project, Markdown, Dataset, Image, Dir |
| `is_deleted` | `BOOLEAN` | 默认 False | 逻辑删除标记 |
| `created_at` | `TIMESTAMP` | 自动生成 | 资产登记时间 |
| `updated_at` | `TIMESTAMP` | 自动更新 | 资产属性最后变更时间 |

---

## 4. 第三层：引用层 (Reference Layer)
引用层只负责建立 M:N 的逻辑连线，不关心资产的具体属性。

### 4.1 资产关系连线表 (`asset_link`)
| 字段名 | 类型 | 约束 | 描述说明 |
| :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | 主键, 自增 | 关系记录 ID |
| `source_id` | `BIGINT` | 索引 | 起始资产 ID（关联 `asset_registry.asset_id`） |
| `target_id` | `BIGINT` | 索引 | 目标资产 ID（关联 `asset_registry.asset_id`） |
| `relation_type` | `VARCHAR(50)` | 索引 | **关系语义**：`CONTAINS` (包含), `REFERENCES` (引用), `DERIVED_FROM` (衍生) |

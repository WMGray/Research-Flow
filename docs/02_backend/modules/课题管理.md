# 课题管理模块设计

## 1. 文档目标

本文档定义 Research-Flow 后端中“课题管理”模块的职责边界、资源结构、流程设计、状态模型、任务编排、文档布局、关系管理与接口映射，作为 `Project` 资源实现的直接设计依据。

本文档重点回答：

- `Project` 模块负责什么
- 课题如何从创建到沉淀长期研究上下文
- 六个核心模块如何初始化与维护
- 课题如何关联论文、Knowledge、Dataset、Presentation
- 模块级 LLM / skill 任务如何触发和写回
- 课题内容如何保护已确认的人为修改

本文档不展开：

- 具体表字段类型
- 前端页面交互设计
- 单个 LLM provider 的 SDK 细节

---

## 2. 模块定位

课题管理模块负责平台中的 `Project` 资源管理，是从单篇论文结果走向长期研究空间的核心承接层。

它承担以下职责：

- 创建与维护课题实体
- 初始化六个核心模块
- 管理课题状态与基本信息
- 关联论文并标注关系类型
- 关联 Knowledge、Dataset、Presentation 与实验记录
- 触发 related work、method、experiment、conclusion、manuscript 等模块级任务
- 将阶段性材料沉淀为长期可复用的 Markdown 工作区

它连接：

- 上游：文献管理模块输出的 `Paper`、章节结果、`note.md`、Knowledge、Dataset
- 下游：Presentation、论文写作、投稿 / Rebuttal、录用后处理

---

## 3. 模块边界

### 3.1 负责范围

课题管理模块负责：

- 创建 `Project`
- 生成 `project_slug`
- 初始化六模块 Markdown 工作区
- 维护课题基础信息与状态
- 维护资源关系与引用角色
- 模块级文档读写与版本保护
- 手动触发模块级 skill / LLM 任务
- 汇总阶段性结果供后续写作与汇报使用

### 3.2 不负责范围

课题管理模块不直接负责：

- PDF 下载与论文原始解析
- 论文级 `note.md` 的生成
- 独立的 Knowledge 工作台
- PPT 模板渲染细节
- 投稿平台规则本身

这些能力由其他模块负责，课题管理模块只消费其输出。

## 4. 资源结构

### 4.1 Project 实体

`Project` 是课题级聚合根。

关键字段至少包括：

- `project_id`
- `project_slug`
- `name`
- `status`
- `summary`
- `owner`
- `created_at`
- `updated_at`

### 4.2 六个核心模块

每个课题创建后自动初始化以下六个模块：

- `overview.md`
- `related-work.md`
- `method.md`
- `experiment.md`
- `conclusion.md`
- `manuscript.md`

### 4.3 模块职责

#### `overview`

用于呈现课题总览：

- 研究目标
- 当前状态
- 关键论文
- 关键数据集
- 近期进展
- 下一步计划

#### `related-work`

用于组织与比较关联论文：

- 论文分组
- 方法脉络
- 代表性工作摘要
- 相关工作综述草稿

#### `method`

用于沉淀方法设计：

- 核心假设
- 技术路线
- 模块拆解
- 创新点与风险点

#### `experiment`

用于记录实验设计与结果：

- 实验假设
- 数据集选择
- 指标与 baseline
- 结果记录
- 异常分析

#### `conclusion`

用于沉淀阶段性判断：

- 当前结论
- 尚未解决的问题
- 下一步决策

#### `manuscript`

用于形成论文草稿工作区：

- 章节草稿
- 模块结果汇总
- 已确认片段
- 版本演进

## 5. 目录与文件布局

### 5.1 目录规则

课题资源统一落在：

```text
Projects/{project_slug}/
```

展示名保存在数据库，目录名使用 slug。

### 5.2 建议文件布局

```text
Projects/
  {project_slug}/
    overview.md
    related-work.md
    method.md
    experiment.md
    conclusion.md
    manuscript.md
    assets/
    exports/
    jobs/
```

### 5.3 辅助目录职责

- `assets/`：课题附属图片、表格、引用材料
- `exports/`：阶段性导出物
- `jobs/`：模块任务记录、调试输出或流程摘要

### 5.4 代码包结构

后端代码落在 `backend/core/services/projects/`，当前为 4 文件子包：

```text
backend/core/services/projects/
├── __init__.py          # barrel export：ProjectRepository、DTO、errors、helpers
├── models.py            # ProjectRecord / ProjectDocumentRecord / LinkedPaperRecord + errors
├── repository.py        # ProjectRepository：CRUD、文档管理、论文关联
└── serialization.py     # records_to_dicts / record_to_dict / document_to_dict
```

P1 模块任务运行时（related_work / method / experiment / conclusion / manuscript / overview）落地时将新增 `tasks/` 子包，参照 Papers 的 `refine/`、`note/` 模式。

## 6. 关联模型

### 6.1 课题与论文关系

课题需要支持与已入库论文建立显式关联，关系类型至少包括：

- `related_work`
- `baseline`
- `inspiration`
- `method_reference`
- `experiment_reference`

### 6.2 课题与其他资源关系

课题还应支持关联：

- `Knowledge`
- `Dataset`
- `Presentation`
- 实验记录

### 6.3 关系要求

关系模型应保留：

- 关系类型
- 来源记录
- 创建时间
- 排序或优先级
- 备注信息

这样后续：

- `related-work.md` 可以按关系类型自动汇总
- `method.md` 可以聚合 `method_reference`
- `experiment.md` 可以聚合 `baseline` 与 `experiment_reference`

## 7. 核心流程总览

课题管理模块标准主链路如下：

1. 创建课题
2. 生成 `project_slug`
3. 初始化六模块文档
4. 写入基础元数据与状态
5. 关联论文与其他资源
6. 手动触发模块级任务
7. 将任务结果写回模块文档
8. 保留来源追溯、版本与锁定状态
9. 为后续 Presentation 与 manuscript 提供稳定上下文

## 8. 详细流程设计

### 8.1 创建课题

输入：

- 课题名称
- 可选研究方向
- 可选简介

执行：

- 生成稳定的 `project_slug`
- 创建 `Project` 实体
- 创建目录
- 初始化六个 Markdown 文件
- 写入初始 front matter 或基础模板

输出：

- `project_id`
- `project_slug`
- 六模块工作区

### 8.2 六模块初始化

初始化目标不是直接生成完整内容，而是先建立稳定骨架。

每个模块初始化时建议至少包含：

- 标题
- 更新时间
- 当前来源列表占位
- 待补充说明

这样可以保证：

- 用户可以立即人工编辑
- 后续 skill 任务可以定向写回
- API 层能统一处理模块读写

### 8.3 关联论文

用户可将任意已入库论文关联到课题，并标注关系类型。

执行：

- 校验 `paper_id`
- 校验关系类型
- 创建显式关系记录
- 更新 `overview` 统计信息

输出：

- 课题的论文关系集合
- 可供模块任务使用的论文输入池

### 8.4 关联 Knowledge / Dataset / Presentation

执行：

- 建立跨资源关系
- 标记引用角色
- 允许后续模块按资源类别筛选输入

这样：

- `method` 可优先使用相关 `Knowledge`
- `experiment` 可优先读取 `Dataset`
- `Presentation` 可回溯到课题模块与论文输入

### 8.5 模块级任务触发

课题管理模块的自动化任务以“模块级手动触发”为主。

首版至少支持：

- related work 整理
- method 草稿生成
- experiment 建议
- result / conclusion 汇总
- manuscript 汇总生成

执行方式：

- 通过 `Agno`
- 使用 `SkillBinding`
- 绑定 `AgentProfile` 与 Skill runtime instruction

### 8.6 结果写回

任务完成后，结果写回对应模块 Markdown。

写回策略：

- 首次生成：写入完整草稿
- 再次生成：按区块或片段增量更新
- 已锁定片段：默认不覆盖

### 8.7 overview 自动汇总

`overview.md` 是最适合做自动汇总的模块。

建议自动汇总（系统托管区块，`managed="true"`）：

```md
<!-- RF:BLOCK_START id="overview_stats" managed="true" version="1" -->
| 指标 | 当前值 |
|------|--------|
| 已关联论文 | 12 |
| 关联数据集 | 3 |
| 关联 Knowledge | 8 |
| 最近任务 | project_generate_method (2026-04-23) |
| 课题状态 | researching |
<!-- RF:BLOCK_END id="overview_stats" -->
```

以下为用户自由补充区域（在托管区块之外）：

- 研究目标与当前进展总结
- 关键发现与决策记录
- 下一步计划

规则：

- 系统只更新 `managed="true"` 的区块
- 托管区块之间的用户自由文本默认不被覆盖
- 用户可在任意位置插入自定义内容，包括托管区块之外的 markdown 段落

### 8.8 manuscript 汇总

`manuscript.md` 不是孤立生成，而是消费其他五个模块的结果。

输入来源：

- 读取最新文件内容：`overview.md`、`related-work.md`、`method.md`、`experiment.md`、`conclusion.md`
- 读取关系数据：已关联论文的章节结果与 `note.md`
- 读取 Knowledge / Dataset 关联记录

输入读取方式为"读文件 + 查关系表"，不依赖数据库缓存的文件内容副本。

输出：

- 论文草稿工作区
- 已确认片段与待补全文段

要求：

- 支持锁定已确认内容（`managed="false"` 区块跳过）
- 支持再次汇总时跳过锁定片段

## 9. 状态模型

### 9.1 Project 状态

当前建议至少支持：

- `planning`
- `researching`
- `experimenting`
- `writing`
- `archived`

### 9.2 模块内容状态

建议为模块文档额外维护内容状态：

- `empty`
- `draft_generated`
- `user_modified`
- `merged`
- `locked_partial`
- `conflict_pending`

这样可以区分：

- 模块是否生成过
- 是否已经有人工编辑
- 是否有锁定片段
- 是否存在冲突待处理

## 10. 文档保护与版本策略

### 10.1 基本原则

课题下的自动生成内容不能无提示覆盖人为内容。

### 10.2 建议机制

建议模块文档维护：

- `base_version`
- `updated_at`
- `last_generated_at`
- `last_user_modified_at`
- `is_locked`

### 10.3 锁定策略与标记格式

复用文献管理模块中定义的区块标记格式：

```md
<!-- RF:BLOCK_START id="method_core_pipeline" managed="true" version="1" -->
内容...
<!-- RF:BLOCK_END id="method_core_pipeline" -->
```

规则：

- `managed="true"` 的区块可由系统自动生成覆盖
- `managed="false"` 或标记外的用户内容默认不被覆盖
- 再次生成时只更新 `managed="true"` 区块

锁定适用场景：

- `manuscript` 中已确认段落
- `method` 中已确认方法描述
- `conclusion` 中已确认阶段判断

锁定后：

- 自动任务默认跳过 `managed="false"` 区块
- 用户可通过修改标记属性解除锁定

注意：课题模块文档的"片段"粒度与论文模块中的 `note.md` 一致，以 `RF:BLOCK_START/END` 包围的连续内容为一个单位。

## 11. 任务系统设计

### 11.1 任务类型

建议至少定义以下任务：

- `project_generate_related_work`
- `project_generate_method`
- `project_generate_experiment_plan`
- `project_generate_conclusion`
- `project_generate_manuscript`
- `project_refresh_overview`

### 11.2 并发原则

建议遵循：

- 不同课题之间允许并发
- 同一课题内不同模块的任务允许并发（写的是不同文件，互不冲突）
- 同一模块同一时刻只允许一个写任务

原因：

- 不同模块对应独立的 Markdown 文件，并发写入不会冲突
- SQLite 写锁通过 Celery Worker 有限并发数（建议 `--concurrency=4`）控制
- 只有同一模块文件上的多个写任务才需要串行

### 11.3 资源键建议

使用两级资源键区分粒度：

- `project:{project_id}`：课题级协调，用于跨模块的状态一致性检查
- `project:{project_id}:module:{role}`：模块级锁，同一模块同时只允许一个写任务

不同模块（role 不同）的任务使用不同的资源键，天然可以并发。

## 12. LLM / Skill 设计

### 12.1 接入原则

课题管理模块不直接绑定具体 provider，而统一依赖：

- `Agno`
- `AgentProfile`
- `SkillPackage`
- `SkillBinding`
- `LlmProvider`

### 12.2 建议 skill 场景

建议至少存在：

- `project_related_work`
- `project_method_draft`
- `project_experiment_plan`
- `project_conclusion_summary`
- `project_manuscript_compile`

### 12.3 输入组织方式

模块任务输入应来自：

- 关联论文的章节结果
- 关联论文的 `note.md`
- 关联 Knowledge
- 关联 Dataset
- 当前模块现有内容
- 用户补充说明

这样生成结果才不会脱离课题上下文。

## 13. 接口映射

### 13.1 Project 资源接口

- `POST /api/v1/projects`
- `GET /api/v1/projects`
- `GET /api/v1/projects/{project_id}`
- `PATCH /api/v1/projects/{project_id}`
- `DELETE /api/v1/projects/{project_id}`

### 13.2 模块文档接口

- `GET /api/v1/projects/{project_id}/documents/{role}`
- `PUT /api/v1/projects/{project_id}/documents/{role}`

其中 `role` 包括：

- `overview`
- `related_work`
- `method`
- `experiment`
- `conclusion`
- `manuscript`

### 13.3 关系接口

- `POST /api/v1/projects/{project_id}/papers:link`
- `DELETE /api/v1/projects/{project_id}/papers/{paper_id}`

建议后续补充：

- `POST /api/v1/projects/{project_id}/knowledge:link`
- `POST /api/v1/projects/{project_id}/datasets:link`
- `POST /api/v1/projects/{project_id}/presentations:link`

### 13.4 模块任务触发接口

触发模块级 LLM 任务，均为异步模式（创建 Job 后立即返回）：

- `POST /api/v1/projects/{project_id}/generate-related-work`
- `POST /api/v1/projects/{project_id}/generate-method`
- `POST /api/v1/projects/{project_id}/generate-experiment`
- `POST /api/v1/projects/{project_id}/generate-conclusion`
- `POST /api/v1/projects/{project_id}/generate-manuscript`
- `POST /api/v1/projects/{project_id}/refresh-overview`

**请求体**（所有接口共用）：

```json
{
  "focus_instructions": "可选的额外聚焦说明",
  "included_paper_ids": ["paper_id_1"],
  "included_knowledge_ids": ["kn_id_1"],
  "skip_locked_blocks": true
}
```

**返回体**：

```json
{
  "code": "OK",
  "data": {
    "job_id": "uuid",
    "job_type": "project_generate_related_work",
    "status": "queued",
    "created_at": "2026-04-24T10:00:00Z"
  }
}
```

说明：

- `focus_instructions`：用户可附加本次生成的针对性说明（如"重点关注 2025 年以后的对比方法"）
- `included_paper_ids` / `included_knowledge_ids`：限定本次任务的输入范围，默认使用全部已关联资源
- `skip_locked_blocks`：默认 `true`，对 `managed="false"` 的区块不覆盖

## 14. 失败与恢复策略

### 14.1 初始化失败

处理方式：

- 保留 `Project` 记录
- 标记哪些文档未创建成功
- 支持重新初始化缺失模块

### 14.2 模块生成失败

处理方式：

- 标记对应任务失败
- 不覆盖原有模块内容
- 允许重跑

### 14.3 文档版本冲突

处理方式：

- 返回 `DOCUMENT_VERSION_CONFLICT`
- 阻止覆盖
- 要求客户端基于最新版本重试

### 14.4 锁定片段冲突

处理方式：

- 保持原片段不变
- 标记 `conflict_pending`
- 等待人工处理

**P1 — 详细实现方案**

以下为各 P1 缺口的具体实现设计。

**1. Project 删除端点**

`DELETE /api/v1/projects/{project_id}` 实现软删除：

- 设置 `asset_registry.is_deleted = 1`
- 级联标记：该 Project 下的 `biz_doc_layout` 记录标记为 deleted（或通过 asset_registry 软删除机制自动级联）
- 不删除物理文件（`Projects/{project_slug}/` 目录保留）
- 不删除 `asset_link` 记录（保留关系历史）

代码改动：
- `core/services/projects/repository.py`：新增 `delete_project(project_id)` 方法
- `app/api/projects.py`：新增 `@router.delete("/projects/{project_id}")`
- `app/schemas/projects.py`：无新增 schema（204 No Content 响应）

**2. 模块级 LLM 任务触发（6 个接口）**

接口设计见 §13.4。实现架构参照 Papers 的 `refine/runtime.py` + `note/runtime.py` 模式：

新增代码包结构：
```text
backend/core/services/projects/tasks/
├── __init__.py            # barrel export
├── models.py              # task input/output DTOs
├── related_work.py        # run_generate_related_work()
├── method.py              # run_generate_method()
├── experiment.py          # run_generate_experiment()
├── conclusion.py          # run_generate_conclusion()
├── manuscript.py          # run_generate_manuscript()
└── overview.py            # run_refresh_overview()
```

每个 task runtime 的通用流程：

1. 收集输入：
   - 已关联论文的章节结果（`parsed/sections/*.md`）
   - 已关联论文的 `note.md`
   - 当前模块文档已有内容
   - 用户提供的 `focus_instructions`（来自 request body）
2. 渲染后端 skill runtime instruction（通过 `skill_runtime` 按 skill 目录约定读取）
3. 调用 Agno scene runner，由 `llm_registry` 按 feature 路由到模型
4. 写回结果到模块文档（使用 `RF:BLOCK_START/END` 标记，`managed="true"`）
5. 创建 Job 记录和 pipeline_run 记录

**Agno scene 与 Skill 设计**

新增 6 个后端运行时能力。课题模块后续应沿用 Paper 的 skill 包结构：`skills/{project-skill}/SKILL.md` + `agents/openai.yaml` + `references/runtime-instructions.md`，不再新增 standalone prompt 文件。

**3. overview 自动汇总**

`overview.md` 是最适合自动维护的模块。实现方式：

- 在 `refresh-overview` 任务中，读取关联资源统计（论文数、Knowledge 数、Dataset 数、最近 Job）
- 渲染为 Markdown 表格，写入 `managed="true"` 区块
- 用户自由补充内容在托管区块之外

**4. manuscript 汇总**

`manuscript` 消费其他五个模块的结果：

1. 读取 `overview.md`、`related-work.md`、`method.md`、`experiment.md`、`conclusion.md` 当前内容
2. 读取已关联论文的章节结果和 `note.md`
3. 渲染 prompt（输入为多模块内容 + 论文材料）
4. LLM 生成论文草稿章节
5. 写回 `manuscript.md`，尊重已锁定的 `managed="false"` 区块

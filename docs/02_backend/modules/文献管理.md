# 文献管理模块设计

## 1. 文档目标

本文档定义 Research-Flow 后端中“文献管理”模块的职责边界、核心流程、状态模型、任务编排、存储布局、LLM / skill 接入方式与接口映射，作为 `Paper` 主链路实现的直接设计依据。

本文档重点回答：

- `Paper` 模块具体负责什么
- 论文从导入到入库的完整链路怎么跑
- 哪些步骤同步执行，哪些步骤异步执行
- 人工审查如何进入和退出流程
- `note.md` 如何增量更新
- Knowledge / Dataset 如何从文献链路中衍生
- 模块与 `Job`、`Agno`、`SQLite`、文件系统如何协作

本文档不展开：

- 具体数据库字段类型
- 前端页面交互
- 某个 provider 的 SDK 细节

---

## 2. 模块定位

文献管理模块是整个后端的主入口模块，承担以下职责：

- 创建论文记录
- 解析 DOI / URL / 标题对应的论文元数据
- 在仅有元信息时即完成数据库入库
- 在仅有元信息时即可供 `Project` 关联
- 下载或挂接 `paper.pdf`
- 生成原始解析结果
- 调用 LLM 执行优化解析
- 进入人工审查与确认节点
- 进行章节切分
- 生成与增量更新 `note.md`
- 抽取 `Knowledge` 与 `Dataset`
- 向 `Project` 模块输出可关联的论文资产

它是后续 `Project`、`Knowledge`、`Dataset`、`Presentation` 的前置基础。

## 3. 模块边界

### 3.1 负责范围

文献管理模块负责：

- 单篇论文导入
- 分类归属与 slug 目录初始化
- 文档与 PDF 的文件侧落盘
- 任务创建、流程推进与状态写回
- 解析中间产物管理
- `note.md` 生命周期管理
- 面向论文的结构化抽取入口

### 3.2 不负责范围

文献管理模块不直接负责：

- 课题六模块生成与维护
- 汇报草稿生成
- 会议信息管理
- 论文推荐候选池维护
- Zotero 深度同步

这些能力只消费文献管理模块的输出结果。

## 5. 资源与目录布局

### 5.1 目录规则

文献资源统一落在：

```text
Papers/{direction_slug}/{sub_direction_slug}/{paper_slug}/
```

目录名采用 slug，展示名保存在数据库。

### 5.2 建议文件布局

```text
Papers/
  {direction_slug}/
    {sub_direction_slug}/
      {paper_slug}/
        paper.pdf
        note.md
        metadata.json
        parsed/
          raw.md
          refined.md
          sections/
            related_work.md
            method.md
            experiment.md
            conclusion.md
        extracted/
          knowledge.json
          datasets.json
        jobs/
          history.json
```

### 5.3 文件职责

- `paper.pdf`：论文原始 PDF
- `metadata.json`：文件侧元数据缓存，允许长期保留
- `parsed/raw.md`：解析器原始输出
- `parsed/refined.md`：LLM 优化后文本，也是人工审查直接编辑的文件
- `parsed/sections/*.md`：章节切分结果
- `note.md`：论文统一研究笔记

## 7. 流程总览

文献管理模块的标准主链路如下：

1. 创建 `Paper`
2. 解析基础元数据
3. 初始化目录与资产映射
4. 进入 `metadata_ready`，此时即可落库并供 `Project` 关联
5. 下载或挂接 `paper.pdf`
6. 执行 PDF 原始解析，保存 MinerU `full.md` 为 `parsed/raw.md`
7. 执行 LLM 优化解析，对 `raw.md` 做诊断、局部 patch 修复与验证，生成 `parsed/refined.md`
8. 进入人工审查，确认 `refined.md` 与低置信度 review items
9. 用户确认后执行章节切分
10. 生成或增量更新 `note.md`
11. 抽取 `Knowledge`
12. 抽取 `Dataset`
13. 标记文献链路完成

## 8. 详细流程设计

### 8.1 创建论文记录

输入：

- DOI / URL / 标题之一
- 可选分类
- 可选来源类型

执行：

- 按“用户提供什么就优先解析什么”的原则执行：
  - 提供 DOI：优先按 DOI 解析
  - 提供 URL：优先按 URL 解析
  - 提供标题：优先按标题解析
  - 提供本地 PDF：优先挂接本地文件
- 创建 `Paper` 记录
- 生成稳定的 `paper_slug`
- 建立目录与资产索引
- 写入 `paper_stage = metadata_ready`

输出：

- `paper_id`
- `paper_slug`
- 初始资源记录
- 可被 `Project` 立即关联的论文实体

### 8.2 元数据解析

如果输入的是 DOI / URL / 标题，系统应先尝试解析元数据。

执行：

- 调用 resolver
- 规范化标题、作者、年份、来源
- 进行重复检测，如 DOI 去重

输出：

- 标准化论文元数据
- 是否命中已有资源

说明：

- 元数据一旦可用，论文就已经是合法的资源对象
- 后续 `pdf_url`、本地路径、解析产物都可以分阶段补齐

### 8.3 PDF 获取

PDF 获取阶段支持三类模式：

- `attached_local_pdf`：`pdf_url` 或 `source_url` 指向本地 PDF 时，直接复制或挂接为 `paper.pdf`。
- `gpaper`：生产或真实联网测试环境设置 `RFLOW_ENABLE_NETWORK_PAPER_DOWNLOAD=1` 后，通过 `PaperDownloadService` / gPaper 获取 PDF。
- `metadata_stub`：开发环境无网络或未启用真实下载时生成最小 PDF stub，使本地 smoke 测试可覆盖后续状态机。

下载任务完成后更新：

- `download_status`
- 资产索引
- 文件物理路径

真实生产下载必须显式启用联网开关，并在环境变量中配置 gPaper 所需凭证或邮箱，不把凭证写入仓库。

### 8.4 原始解析

PDF 到 Markdown 的首次转换由解析器完成。parse 阶段只产出 `parsed/raw.md`，不做总结、不直接拆 section、不让 LLM 重写整篇论文。

解析优先级：

1. 优先复用本地已有 MinerU Markdown：
   - `parsed/mineru/full.md`
   - `mineru/full.md`
   - `parsed/full.md`
   - `full.md`
2. 如果存在 `paper.pdf` 且 MinerU token 已配置，调用 `PDFParserService.extract_raw_markdown()`。
3. 否则生成 `metadata_fallback` raw markdown，保证开发环境可跑通状态机。

执行结果：

- 生成 `parsed/raw.md`
- `parsed/raw.md` 对应 MinerU 的 `full.md`，作为后续 LLM 优化解析的主输入
- 写入 `parse_status`
- 记录对应 `Job`

### 8.5 LLM 优化解析

原始解析结果通常来自 MinerU `full.md`，它已经具备较完整的论文文本，但仍可能存在：

- 格式紊乱
- 标题层级不稳定
- 表格公式断裂
- 章节识别不完整
- 图表 caption 混入正文
- 阅读顺序或章节片段错乱

因此需要调用 LLM 执行优化解析。

当前目标不是让 LLM 自由重写全文，而是使用“诊断 + patch + 验证”的受约束 workflow：

```text
parsed/raw.md
  -> parsed/refine/diagnosis.json
  -> parsed/refine/patches.json
  -> parsed/refine/patch_apply_report.json
  -> parsed/refine/prompt_context.json
  -> parsed/refine/verify.json
  -> parsed/refined.md
```

执行方式：

- 通过 `Agno` 调用 `paper_refine_parse` 场景
- 由 `SkillBinding` 绑定 agent profile、prompt template、可用工具
- `diagnose` 阶段读取带行号的 `raw.md`，输出结构化问题清单，不修改正文
- `repair` 阶段只针对问题 span 输出 patch，不输出整篇论文
- `apply` 阶段由程序应用高置信度 patch，低置信度问题写入 review items
- `verify` 阶段检查 citation、数字、公式、表格、caption、标题树和文本长度是否异常变化
- 接口保留 `instruction` 输入口，用于附加本次 refine 的人工要求
- LLM 控制 JSON 解析失败时，进入 warning/no-op：保留 `raw.md`、不应用不可信 patch、写入 warning artifacts，并允许后续 split / summarize 在明确记录风险后继续

输出：

- 生成 `parsed/refined.md`
- 生成 `parsed/refine/*.json` 诊断、patch、验证和待审查报告
- 更新 `refine_status`
- 记录本次 `llm_run_id`

细化设计见 [../services/pdf-parser.md](../services/pdf-parser.md)。

### 8.6 人工审查

人工审查直接复用 `parsed/refined.md`，不创建额外的审查文件。

进入方式：

- `refine-parse` 完成后
- 手动触发 `submit-review`

状态：

- `Job.status = waiting_review`
- `Paper.review_status = waiting_review`

用户行为：

- 读取 `parsed/refined.md`
- 直接修改内容
- 完成后点击确认

确认后：

- `Job.status = waiting_confirm` -> 新建后续任务
- `Paper.review_status = confirmed`

### 8.7 章节切分

输入：

- 经人工确认后的 `parsed/refined.md`

输出：

- `parsed/sections/*.md`
- 四类 canonical sections 的结构化结果
- 章节级索引信息

要求：

- 章节切分只保留四类：
  - `related_work`
  - `method`
  - `experiment`
  - `conclusion`
- 不额外引入 `other`
- 先基于解析结构与标题层级做程序切分候选
- 再由 LLM 做章节归一化与纠偏
- 允许原文中直接以方法名作为章节标题
- 缺失章节允许为空，但要可识别
- 支持后续 Knowledge / note / Dataset 使用章节作为来源定位

说明：

- 如果不使用 LLM，仅靠解析结果直接程序划分，准确率不稳定
- 因此首版建议采用“规则候选 + LLM 归一化”的混合方案

### 8.8 note 生成与增量更新

`note.md` 是论文级统一研究笔记。

首版建议至少包含这些区块：

- Research Question
- Core Method
- Main Contributions
- Experiment Summary
- Limitations

建议的区块标记格式：

```md
<!-- RF:BLOCK_START id="research_question" managed="true" version="1" -->
内容...
<!-- RF:BLOCK_END id="research_question" -->
```

规则：

- 系统只更新 `managed="true"` 的区块
- 区块外的用户内容默认不覆盖
- 已锁定或冲突区块只读或待人工处理

系统行为：

- 首次生成时创建完整 `note.md`
- 再次生成时进行区块级增量合并

用户行为：

- 允许直接编辑 `note.md`

系统必须准确识别状态：

- `clean_generated`
- `user_modified`
- `merged`
- `conflict_pending`

识别依据：

- 文档整体 hash
- 系统托管区块 hash
- 最近一次系统生成时间
- 最近一次用户修改时间
- 最近一次合并结果

### 8.9 Knowledge 抽取

Knowledge 抽取由 `Agno` 执行 skill 化流程。

输入：

- 章节化论文内容
- 文献元数据
- note.md（已生成的论文笔记）
- 抽取目标配置

输出字段（对应 `biz_knowledge` 表，表已落地于 2026-04-27）：

- `title`
- `knowledge_type`
- `summary_zh`
- `original_text_en`
- `citation_marker`
- `category_label`
- `research_field`
- `source_paper_id`
- `source_section`
- `source_locator`
- `evidence_text`
- `confidence_score`
- `review_status`
- `llm_run_id`

落库规则：

- 仅当 `confidence_score >= 0.6` 时允许入库
- 初始 `review_status = pending_review`
- 后续允许人工审核改为 `approved`、`rejected` 或 `edited`

溯源规则：

- 每条 Knowledge 只记录其来源论文、来源章节、原文表述和原文中的引用标记。
- 不强制追溯“最早提出该观点/定义的论文”。
- 如果原文写作中出现 `[32]` 这类引用标记，则将 `[32]` 存入 `citation_marker`，表示这是来源论文 reference list 中的第 32 个参考文献。

当前状态（2026-04-27）：`biz_knowledge` 表已落地，LLM prompt（`paper_extract_knowledge.md`）和入库逻辑待实现。

### 8.10 Dataset 抽取

`Dataset` 不是”单篇论文里的一个字段”，而是聚合多个论文的资源实体。

因此文献管理模块中的 Dataset 抽取应分两步：

1. 从单篇论文中抽取 dataset mentions
2. 将 mention 归一化到已有 `Dataset`，或创建新的 `Dataset`

Dataset 抽取与 Knowledge 抽取共享同一份章节化前处理结果，但不建议共用同一个 skill。

推荐方案：

- 共享前处理 workflow
- 独立的 `paper_extract_knowledge` skill
- 独立的 `paper_extract_datasets` skill

原因：

- `Knowledge` 更偏语义抽取
- `Dataset` 更偏实体识别、归一化与聚合
- 两者共享输入，但不适合混成一个大 prompt

首版至少抽取（对应 `biz_dataset` 表，表已落地于 2026-04-27）：

- 数据集名称
- 任务类型
- 数据领域
- 数据规模
- 来源类型
- 使用场景
- 相关论文
- 指标或实验摘要

当前状态（2026-04-27）：`biz_dataset` 表已落地，LLM prompt（`paper_extract_datasets.md`）和实体归一化逻辑待实现。

## 9. 状态模型

### 9.1 Paper 阶段状态

建议使用 `paper_stage` 表示论文当前所处的串行阶段：

- `metadata_ready`
- `downloaded`
- `parsed`
- `refined`
- `review_confirmed`
- `sectioned`
- `noted`
- `knowledge_extracted`
- `dataset_extracted`
- `completed`
- `error`

说明：

- `paper_stage` 用于表达论文链路推进到了哪一阶段
- `job.status` 用于表达某次任务实例当前是否在运行、等待审查或失败
- `Project` 关联门槛是 `metadata_ready`，而不是必须等到 PDF、解析或 `note` 完成

### 9.2 子状态

为了记录更细粒度的运行明细，仍建议保留分状态字段：

- `download_status`
- `parse_status`
- `refine_status`
- `review_status`
- `note_status`

这样前端可以更细粒度地展示流程进度。

## 10. 任务系统设计

### 10.1 任务类型

建议文献管理模块至少定义以下任务：

- `paper_download`
- `paper_parse`
- `paper_refine_parse`
- `paper_split_sections`
- `paper_generate_note`
- `paper_extract_knowledge`
- `paper_extract_datasets`

### 10.2 并发策略

文献管理模块遵循：

- 不同论文之间允许并发
- 同类型任务在不同论文上允许并发
- 不同类型任务在不同论文上允许并发
- 同一论文上的冲突任务必须串行

示例：

- `paper_A.download` 与 `paper_B.download` 可以并发
- `paper_A.download` 与 `paper_B.parse` 可以并发
- `paper_A.parse` 与 `paper_A.refine` 不可并发
- `paper_A.refine` 与 `paper_A.generate_note` 不可并发

### 10.3 Celery 下的并发策略

文献管理模块的异步任务通过 `Celery` + `Redis` 执行，遵循架构概览中的进程模型：

- `FastAPI` 接收请求，创建 `biz_job` 记录，调用 `.delay()` 投递任务到 Redis
- `Celery Worker` 消费并执行任务，Worker 内部通过 `Agno` 调用 LLM
- `Celery Beat` 负责定时任务（如每日论文推送）

SQLite 写锁控制：

- Worker 并发数建议 `--concurrency=4` 以内
- 同一论文的冲突任务由 `resource_key`（`paper:{paper_id}`）在业务层保证串行
- 不同论文的任务可在多个 Worker 间并发执行

这样既能支持多论文同时处理，又能避免 SQLite 写锁冲突。

### 10.4 Checkpoint 机制

人工审查节点是明确的 workflow checkpoint。

调度器应支持：

- 任务进入 `waiting_review`
- 等待用户更新 `refined.md`
- 用户调用确认接口后恢复流程

## 11. LLM / Skill 设计

### 11.1 使用原则

文献管理模块中的 LLM 调用不直接绑定某个平台 SDK，而是统一通过：

- `Agno`
- `AgentProfile`
- `PromptTemplate`
- `SkillBinding`
- `LlmProvider`

### 11.2 建议 skill 划分

建议至少存在以下 scene / skill：

- `paper_refine_parse`
- `paper_split_sections`
- `paper_generate_note`
- `paper_extract_knowledge`
- `paper_extract_datasets`

### 11.3 SkillBinding 作用

`SkillBinding` 负责表达：

- 某个能力场景属于哪个 skill
- 使用哪个 `AgentProfile`
- 使用哪个 `PromptTemplate`
- 可挂哪些工具

文献管理模块不直接关心具体 provider，只关心 scene 能否执行。

## 13. 失败与恢复策略

### 13.1 下载失败

处理方式：

- 标记 `download_status = failed`
- 保留论文记录
- 支持重新创建下载任务

### 13.2 解析失败

处理方式：

- 标记对应阶段失败
- 不删除已有 PDF
- 允许重跑 parse / refine

### 13.3 人工审查中断

处理方式：

- 保持 `waiting_review`
- 保留 `parsed/refined.md`
- 用户可稍后继续修改并确认

### 13.4 note 合并冲突

处理方式：

- 标记 `note_status = conflict_pending`
- 不自动覆盖用户内容
- 等待用户手动处理或未来提供冲突解决工具

# 文献管理模块设计

## 1. 文档目标

本文档定义 Research-Flow 后端中“文献管理”模块的职责边界、核心流程、状态模型、任务编排、存储布局、LLM / skill 接入方式与接口映射，作为 `Paper` 主链路实现的直接设计依据。

本文档重点回答：

- `Paper` 模块具体负责什么
- 论文从导入到入库的完整链路怎么跑
- 哪些步骤同步执行，哪些步骤异步执行
- 人工审查如何进入和退出流程
- `note.md` 如何增量更新
- Knowledge / Dataset 如何从文献链路中衍生
- 模块与 `Job`、`Agno`、`SQLite`、文件系统如何协作

本文档不展开：

- 具体数据库字段类型
- 前端页面交互
- 某个 provider 的 SDK 细节

---

## 2. 模块定位

文献管理模块是整个后端的主入口模块，承担以下职责：

- 创建论文记录
- 解析 DOI / URL / 标题对应的论文元数据
- 在仅有元信息时即完成数据库入库
- 在仅有元信息时即可供 `Project` 关联
- 下载或挂接 `paper.pdf`
- 生成原始解析结果
- 调用 LLM 执行优化解析
- 进入人工审查与确认节点
- 进行章节切分
- 生成与增量更新 `note.md`
- 抽取 `Knowledge` 与 `Dataset`
- 向 `Project` 模块输出可关联的论文资产

它是后续 `Project`、`Knowledge`、`Dataset`、`Presentation` 的前置基础。

---

## 3. 模块边界

### 3.1 负责范围

文献管理模块负责：

- 单篇论文导入
- 分类归属与 slug 目录初始化
- 文档与 PDF 的文件侧落盘
- 任务创建、流程推进与状态写回
- 解析中间产物管理
- `note.md` 生命周期管理
- 面向论文的结构化抽取入口

### 3.2 不负责范围

文献管理模块不直接负责：

- 课题六模块生成与维护
- 汇报草稿生成
- 会议信息管理
- 论文推荐候选池维护
- Zotero 深度同步

这些能力只消费文献管理模块的输出结果。

## 5. 资源与目录布局

### 5.1 目录规则

文献资源统一落在：

```text
Papers/{direction_slug}/{sub_direction_slug}/{paper_slug}/
```

目录名采用 slug，展示名保存在数据库。

### 5.2 建议文件布局

```text
Papers/
  {direction_slug}/
    {sub_direction_slug}/
      {paper_slug}/
        paper.pdf
        note.md
        metadata.json
        parsed/
          raw.md
          refined.md
          sections/
            related_work.md
            method.md
            experiment.md
            conclusion.md
        extracted/
          knowledge.json
          datasets.json
        jobs/
          history.json
```

### 5.3 文件职责

- `paper.pdf`：论文原始 PDF
- `metadata.json`：文件侧元数据缓存，允许长期保留
- `parsed/raw.md`：解析器原始输出
- `parsed/refined.md`：LLM 优化后文本，也是人工审查直接编辑的文件
- `parsed/sections/*.md`：章节切分结果
- `note.md`：论文统一研究笔记

## 7. 流程总览

文献管理模块的标准主链路如下：

1. 创建 `Paper`
2. 解析基础元数据
3. 初始化目录与资产映射
4. 进入 `metadata_ready`，此时即可落库并供 `Project` 关联
5. 下载或挂接 `paper.pdf`
6. 执行 PDF 原始解析，保存 MinerU `full.md` 为 `parsed/raw.md`
7. 执行 LLM 优化解析，对 `raw.md` 做诊断、局部 patch 修复与验证，生成 `parsed/refined.md`
8. 进入人工审查，确认 `refined.md` 与低置信度 review items
9. 用户确认后执行章节切分
10. 生成或增量更新 `note.md`
11. 抽取 `Knowledge`
12. 抽取 `Dataset`
13. 标记文献链路完成

## 8. 详细流程设计

### 8.1 创建论文记录

输入：

- DOI / URL / 标题之一
- 可选分类
- 可选来源类型

执行：

- 按“用户提供什么就优先解析什么”的原则执行：
  - 提供 DOI：优先按 DOI 解析
  - 提供 URL：优先按 URL 解析
  - 提供标题：优先按标题解析
  - 提供本地 PDF：优先挂接本地文件
- 创建 `Paper` 记录
- 生成稳定的 `paper_slug`
- 建立目录与资产索引
- 写入 `paper_stage = metadata_ready`

输出：

- `paper_id`
- `paper_slug`
- 初始资源记录
- 可被 `Project` 立即关联的论文实体

### 8.2 元数据解析

如果输入的是 DOI / URL / 标题，系统应先尝试解析元数据。

执行：

- 调用 resolver
- 规范化标题、作者、年份、来源
- 进行重复检测，如 DOI 去重

输出：

- 标准化论文元数据
- 是否命中已有资源

说明：

- 元数据一旦可用，论文就已经是合法的资源对象
- 后续 `pdf_url`、本地路径、解析产物都可以分阶段补齐

### 8.3 PDF 获取

PDF 获取阶段支持三类模式：

- `attached_local_pdf`：`pdf_url` 或 `source_url` 指向本地 PDF 时，直接复制或挂接为 `paper.pdf`。
- `gpaper`：生产或真实联网测试环境设置 `RFLOW_ENABLE_NETWORK_PAPER_DOWNLOAD=1` 后，通过 `PaperDownloadService` / gPaper 获取 PDF。
- `metadata_stub`：开发环境无网络或未启用真实下载时生成最小 PDF stub，使本地 smoke 测试可覆盖后续状态机。

下载任务完成后更新：

- `download_status`
- 资产索引
- 文件物理路径

真实生产下载必须显式启用联网开关，并在环境变量中配置 gPaper 所需凭证或邮箱，不把凭证写入仓库。

### 8.4 原始解析

PDF 到 Markdown 的首次转换由解析器完成。parse 阶段只产出 `parsed/raw.md`，不做总结、不直接拆 section、不让 LLM 重写整篇论文。

解析优先级：

1. 优先复用本地已有 MinerU Markdown：
   - `parsed/mineru/full.md`
   - `mineru/full.md`
   - `parsed/full.md`
   - `full.md`
2. 如果存在 `paper.pdf` 且 MinerU token 已配置，调用 `PDFParserService.extract_raw_markdown()`。
3. 否则生成 `metadata_fallback` raw markdown，保证开发环境可跑通状态机。

执行结果：

- 生成 `parsed/raw.md`
- `parsed/raw.md` 对应 MinerU 的 `full.md`，作为后续 LLM 优化解析的主输入
- 写入 `parse_status`
- 记录对应 `Job`

### 8.5 LLM 优化解析

原始解析结果通常来自 MinerU `full.md`，它已经具备较完整的论文文本，但仍可能存在：

- 格式紊乱
- 标题层级不稳定
- 表格公式断裂
- 章节识别不完整
- 图表 caption 混入正文
- 阅读顺序或章节片段错乱

因此需要调用 LLM 执行优化解析。

当前目标不是让 LLM 自由重写全文，而是使用“诊断 + patch + 验证”的受约束 workflow：

```text
parsed/raw.md
  -> parsed/refine/diagnosis.json
  -> parsed/refine/patches.json
  -> parsed/refine/patch_apply_report.json
  -> parsed/refine/skill_context.json
  -> parsed/refine/verify.json
  -> parsed/refined.md
```

执行方式：

- 通过 `Agno` 调用 `paper-refine-parse` skill 对应场景
- 由 `skill_runtime.py` 解析 `paper_refine_parse.*` instruction key，并从 `skills/paper-refine-parse/references/runtime-instructions.md` 读取模型指令
- 由 feature routing 绑定 provider、model、temperature 与可用工具
- `diagnose` 阶段读取带行号的 `raw.md`，输出结构化问题清单，不修改正文
- `repair` 阶段只针对问题 span 输出 patch，不输出整篇论文
- `apply` 阶段由程序应用高置信度 patch，低置信度问题写入 review items
- `verify` 阶段检查 citation、数字、公式、表格、caption、标题树和文本长度是否异常变化
- 接口保留 `instruction` 输入口，用于附加本次 refine 的人工要求
- LLM 控制 JSON 解析失败时，进入 warning/no-op：保留 `raw.md`、不应用不可信 patch、写入 warning artifacts，并允许后续 split / summarize 在明确记录风险后继续

输出：

- 生成 `parsed/refined.md`
- 生成 `parsed/refine/*.json` 诊断、patch、验证和待审查报告
- 更新 `refine_status`
- 记录本次 `llm_run_id`

细化设计见 [../services/pdf-parser.md](../services/pdf-parser.md)。

### 8.6 人工审查

人工审查直接复用 `parsed/refined.md`，不创建额外的审查文件。

进入方式：

- `refine-parse` 完成后
- 手动触发 `submit-review`

状态：

- `Job.status = waiting_review`
- `Paper.review_status = waiting_review`

用户行为：

- 读取 `parsed/refined.md`
- 直接修改内容
- 完成后点击确认

确认后：

- `Job.status = waiting_confirm` -> 新建后续任务
- `Paper.review_status = confirmed`

### 8.7 章节切分

输入：

- 经人工确认后的 `parsed/refined.md`

输出：

- `parsed/sections/*.md`
- 五类 canonical sections 的结构化结果
- 章节级索引信息

要求：

- 章节切分只保留五类：
  - `related_work`
  - `method`
  - `experiment`
  - `appendix`
  - `conclusion`
- 不额外引入 `other`
- 先基于解析结构与标题层级做程序切分候选
- 再由 LLM 做章节归一化与纠偏
- 允许原文中直接以方法名作为章节标题
- 缺失章节允许为空，但要可识别
- 支持后续 Knowledge / note / Dataset 使用章节作为来源定位

#### 实现方案：规则候选 + LLM 归一化

当前实现（2026-04-27）基于 heading alias 启发式规则直接切分。目标方案分两阶段：

**Stage 1 — 规则候选（当前已实现）**

`core/services/papers/parse/sections.py` + `core/services/papers/split/heuristics.py`：
- 基于 heading 文本匹配（`related work`、`method`、`experiment`、`conclusion` 及其变体）
- 处理 MinerU 层级错乱（父/子 heading 同级时的编号推断）
- 排除 References、Acknowledgments、affiliations
- 保留 Appendix 内容归入对应 canonical section
- 输出 `parsed/sections/{related_work,method,experiment,appendix,conclusion}.md`

**Stage 2 — LLM 归一化（部分实现）**

当规则候选的 confidence 不足时（如 section 边界模糊、多个 candidate heading 竞争），进入 LLM 归一化：

1. 构建 `section_outline`（带行号的 heading tree）
2. 调用 `paper-sectioning` skill runtime instruction：`paper_sectioning.default`
   - 指令文件：`skills/paper-sectioning/references/runtime-instructions.md`
   - 兼容别名：`paper_section_split.default`
3. LLM 返回 audited line ranges（`start_line`, `end_line`, `section_key`, `confidence`）
4. 后端按 line ranges 复制 refined.md 内容到各 section 文件
5. 对 confidence < 0.65 的 range：回退到规则候选结果
6. LLM 调用失败：完整回退到规则候选结果，pipeline 继续

改动点：
- `core/services/papers/split/runtime.py`：新增 `run_llm_section_split()`，当前已存在 `run_llm_section_split_fallback` 作为基础实现
- 将 `paper-sectioning` 接入 Agno scene runner，并通过 feature routing 选择 provider、model 与 feature
- 扩展 `run_split_sections()` 在规则候选结果置信度低时调用 LLM

**当前差距**：heading alias heuristics 已工作，LLM fallback 已通过 `paper_sectioning.default` 读取 skill runtime instructions 并接入 LLM registry。后续需要把低置信度触发策略、人工审查展示和 split report 前端化补齐。

### 8.8 note 生成与增量更新

`note.md` 是论文级统一研究笔记。

首版包含六个托管区块（由后端运行时模板 `paper_note_generate.default` 定义）：

- `paper_overview` — 文章摘要、标题、作者、年份、会议/期刊、领域定位
- `terminology_guide` — 缩写与术语解释
- `background_motivation` — 深度背景与动机分析
- `experimental_setup` — 实验设置
- `method` — 本文方法
- `experimental_results` — 实验结果

区块标记格式：

```md
<!-- RF:BLOCK_START id=”paper_overview” managed=”true” version=”1” -->
内容...
<!-- RF:BLOCK_END id=”paper_overview” -->
```

规则：

- 系统只更新 `managed=”true”` 的区块
- 区块外的用户内容默认不覆盖
- 已锁定或冲突区块只读或待人工处理

系统行为：

- 首次生成时创建完整 `note.md`
- 再次生成时进行区块级增量合并

用户行为：

- 允许直接编辑 `note.md`

系统必须准确识别状态：

- `clean_generated`
- `user_modified`
- `merged`
- `conflict_pending`

识别依据：

- 文档整体 hash
- 系统托管区块 hash
- 最近一次系统生成时间
- 最近一次用户修改时间
- 最近一次合并结果

这些信号应落入独立的 `biz_note_state` 表（设计见 `docs/02_backend/表设计.md` §4.13）。

#### 实现方案：区块级增量合并

**增量生成流程**

`run_generate_note()` 扩展为：

1. 读取当前 `note.md` 内容
2. 读取 `biz_note_state`（如果不存在则视为首次生成）
3. 根据 `note_status` 决定策略：

| note_status | 策略 |
|-------------|------|
| `empty` | 全新生成全部 6 个 `managed=”true”` 区块 |
| `clean_generated` | 全新生成覆盖全部 6 个区块（用户未修改，安全覆盖） |
| `user_modified` | 逐区块比对 hash：仅覆盖 hash 未变化的 managed 区块；用户已修改的区块保留；缺失区块追加到文末 |
| `merged` | 同 `user_modified`，但额外记录 merge_count |
| `conflict_pending` | 拒绝覆盖，job 返回 failed，要求用户手动处理冲突 |

4. 写入或更新 `note.md`
5. 更新 `biz_note_state`：
   - `doc_hash` = 新全文 hash
   - `managed_block_hashes` = 各 block 新 hash
   - `last_generated_at` = now
   - 如果 note_status 为 `user_modified` 且发生合并，更新 `last_merged_at`

**冲突检测规则**

`conflict_pending` 的产生条件：
- 同一个 managed block 被用户修改过（hash 与系统记录不一致）且 LLM 本次产出了不同的内容
- 即：用户改过的 block 且新生成内容与该 block 当前内容不同 → 标记冲突

用户处理冲突：
- 手动编辑 note.md 解决冲突后，将 `note_status` 改为 `user_modified`
- 下次 generate-note 进入正常合并策略

**`biz_note_state` 表**

见 `docs/02_backend/表设计.md` §4.13。核心字段：
- `paper_id`（FK → biz_paper.asset_id, UNIQUE）
- `doc_hash` — 全文 SHA256
- `managed_block_hashes` — JSON: `{“paper_overview”: “sha256...”, ...}`
- `last_generated_at`
- `last_user_modified_at`
- `last_merged_at`
- `merge_count`
- `conflict_blocks` — JSON array of conflicted block IDs
- `created_at` / `updated_at`

**当前差距**（2026-04-27）：
- `note_status` 字段和基本状态转换已实现
- `RF:BLOCK_START/END` 标记格式和首次生成已实现
- 尚未实现：区块级 hash 追踪、`biz_note_state` 表、增量合并逻辑、冲突检测

### 8.9 Knowledge 抽取

Knowledge 抽取由 `Agno` 执行后端 scene 化流程。

输入：

- 章节化论文内容
- 文献元数据
- note.md（已生成的论文笔记）
- 抽取目标配置

输出字段（对应 `biz_knowledge` 表，表已落地于 2026-04-27）：

- `title`
- `knowledge_type`
- `summary_zh`
- `original_text_en`
- `citation_marker`
- `category_label`
- `research_field`
- `source_paper_id`
- `source_section`
- `source_locator`
- `evidence_text`
- `confidence_score`
- `review_status`
- `llm_run_id`

落库规则：

- 仅当 `confidence_score >= 0.6` 时允许入库
- 初始 `review_status = pending_review`
- 后续允许人工审核改为 `approved`、`rejected` 或 `edited`

溯源规则：

- 每条 Knowledge 只记录其来源论文、来源章节、原文表述和原文中的引用标记。
- 不强制追溯“最早提出该观点/定义的论文”。
- 如果原文写作中出现 `[32]` 这类引用标记，则将 `[32]` 存入 `citation_marker`，表示这是来源论文 reference list 中的第 32 个参考文献。

当前状态（2026-04-27）：`biz_knowledge` 表已落地，`paper-knowledge-mining` skill 与 runtime instructions 已补齐；Agno scene 和入库逻辑待实现。以下为完整实现方案。

#### 实现方案

**8.9.1 Skill runtime 与输出契约**

新建 Paper skill `paper-knowledge-mining`，输入为论文元数据 + sections + note.md，输出结构化 JSON。运行时模型指令放在 `skills/paper-knowledge-mining/references/runtime-instructions.md`，由 `paper_knowledge_mining.default` instruction key 读取。`paper_extract_knowledge.default` 仅作为历史 job name 的兼容别名。

Knowledge 类型采用两级分类：一级类型 `knowledge_type` + 二级类别 `category_label`。

第一大类和第二大类对应 URD §4.4.1 与 §4.4.2：

**第一大类：view（观点/看法/主张）— URD §4.4.1**

`view` 不是对文章的摘要总结，而是提取作者针对领域、方法或问题发表的明确看法、批判或主张，应具有"可引用性"（Citable）。

| category_label | 含义 | 示例 |
|----------------|------|------|
| `field_status` | 领域现状评述：作者对当前研究趋势、现实需求或领域重要性的主观评价 | "作者认为 scalability 是当前阻碍 LLM 落地的最大瓶颈" |
| `method_critique` | 对现有方法的批判：作者对前人工作指出的具体缺陷或失败场景 | "作者指出 prior work 在 low-resource 场景下失效，因为忽略了 XX 因素" |
| `core_insight` | 核心启发/主张：作者在方法论层面提出的核心假说或洞见 | "作者认为引入 auxiliary recurrence 机制才能根本上解决 long-context 退化" |
| `key_conclusion` | 关键性结论：打破常规认知或具有强主导性的实验发现 | "实验证明 XX 特征对鲁棒性的影响比模型规模更具决定性" |

**第二大类：definition（定义与特征）— URD §4.4.2**

`definition` 提取客观的、具有复用价值的领域名词、任务定义、特征描述、指标公式和数据规范。

| category_label | 含义 | 示例 |
|----------------|------|------|
| `concept_term` | 核心概念/术语定义：领域专有名词及其学术解释 | "Chain-of-Thought prompting 的形式化定义" |
| `task_definition` | 任务定义类：研究任务的学术定义、目标、输入输出规范与边界条件 | "Open-Domain QA 的任务定义：给定问题 q，从知识源 K 中检索证据并生成答案 a" |
| `feature_semantics` | 特征定义类：关键特征的物理含义、多模态语义或数学公式表达 | "Prosody feature 包含 pitch contour、energy envelope 和 speaking rate 三个维度" |
| `feature_extraction` | 特征提取类：提取特征所用的具体算法、网络结构或数据处理流程 | "使用 12-layer Transformer Encoder 提取 frame-level 声学特征，hidden dim = 768" |
| `evaluation_metric` | 评估指标定义：性能评估指标的数学计算方式或度量逻辑 | "ROUGE-L 的计算方式为最长公共子序列的 F1 score" |
| `data_annotation_rule` | 数据构建/标注规则：数据采集、清洗或人工标注的具体规范 | "人工标注时，若两个 annotator 的 label 不一致，由第三位 senior annotator 裁定" |

输出 JSON Schema：

```json
{
  "items": [
    {
      "title": "Knowledge 标题（简洁英文）",
      "knowledge_type": "view | definition",
      "category_label": "field_status | method_critique | core_insight | key_conclusion | concept_term | task_definition | feature_semantics | feature_extraction | evaluation_metric | data_annotation_rule",
      "summary_zh": "中文摘要，2-5 句完整表达",
      "original_text_en": "来源论文中的英文原文表述（view 类型必填，definition 类型可选）",
      "citation_marker": "[32] 或 null",
      "research_field": "研究领域，如 NLP / CV / RL",
      "source_section": "related_work | method | experiment | appendix | conclusion",
      "source_locator": "行号范围或段落定位",
      "evidence_text": "支撑该 Knowledge 的上下文原文",
      "confidence_score": 0.85
    }
  ]
}
```

字段约束：

- `knowledge_type` + `category_label` 必须匹配上述两级分类，不可自由发挥。
- `view` 类型：重点是作者的主观论断。`original_text_en` 必填以保留引用准确性。`category_label` 从 `field_status` / `method_critique` / `core_insight` / `key_conclusion` 中选取。
- `definition` 类型：重点是客观知识的准确归纳，涵盖定义、特征、指标和规范。`original_text_en` 可选。`category_label` 从 `concept_term` / `task_definition` / `feature_semantics` / `feature_extraction` / `evaluation_metric` / `data_annotation_rule` 中选取。
- 每条 Knowledge 只需要记录来源论文和原文中的 citation_marker，不追溯"最早提出者"。
- `citation_marker` 是来源论文 reference list 中的编号（如 `[32]`），不是跨论文引用解析。

### 8.10 Dataset 抽取

`Dataset` 不是”单篇论文里的一个字段”，而是聚合多个论文的资源实体。

因此文献管理模块中的 Dataset 抽取应分两步：

1. 从单篇论文中抽取 dataset mentions
2. 将 mention 归一化到已有 `Dataset`，或创建新的 `Dataset`

Dataset 抽取与 Knowledge 抽取共享同一份章节化前处理结果，但不建议共用同一个 skill。

推荐方案：

- 共享前处理 workflow
- 独立的 `paper-knowledge-mining` skill
- 独立的 `paper-dataset-mining` skill

原因：

- `Knowledge` 更偏语义抽取
- `Dataset` 更偏实体识别、归一化与聚合
- 两者共享输入，但不适合混成一个大 prompt

首版至少抽取（对应 `biz_dataset` 表，表已落地于 2026-04-27）：

- 数据集名称
- 任务类型
- 数据领域
- 数据规模
- 来源类型
- 使用场景
- 相关论文
- 指标或实验摘要

**8.10.3 Name Normalization Algorithm**

程序侧归一化函数（不依赖 LLM）：

```python
def normalize_dataset_name(name: str) -> str:
    # 1. 转小写
    # 2. 去除版本号点变体: "coco 2017" vs "coco2017" → "coco_2017"
    # 3. 统一分隔符为下划线
    # 4. 去除尾部版本年份的括号变体: "imagenet (2012)" → "imagenet_2012"
    # 5. 标准缩写映射: "ms coco" → "mscoco", "ms-coco" → "mscoco"
    ...
```

去重匹配逻辑：
1. 精确匹配 `biz_dataset.normalized_name`
2. 对 `aliases_json` 中的每个别名做 normalize 后匹配
3. 匹配成功 → 使用已有 `dataset.asset_id`，只创建 `asset_link(USES)`
4. 匹配失败 → 创建新 `asset_registry`（`asset_type='Dataset'`）→ `biz_dataset` → `asset_link(USES)`

   b. 查询已有 `biz_dataset`（JOIN `normalized_name` 精确匹配 + `aliases_json` LIKE 模糊匹配）
   c. 命中已有 → 创建 `asset_link`（`source_id=paper.asset_id`, `target_id=dataset.asset_id`, `relation_type='USES'`）
   d. 未命中 → 创建 `asset_registry` → `biz_dataset` → `asset_link(USES)`

6. 推进 `paper_stage = 'completed'`
7. 返回 job 记录（`result` 包含 `new_datasets` 和 `linked_datasets` 计数）

## 10. 任务系统设计

### 10.1 任务类型

建议文献管理模块至少定义以下任务：

- `paper_download`
- `paper_parse`
- `paper_refine_parse`
- `paper_split_sections`
- `paper_generate_note`
- `paper_extract_knowledge`
- `paper_extract_datasets`

### 10.2 并发策略

文献管理模块遵循：

- 不同论文之间允许并发
- 同类型任务在不同论文上允许并发
- 不同类型任务在不同论文上允许并发
- 同一论文上的冲突任务必须串行

示例：

- `paper_A.download` 与 `paper_B.download` 可以并发
- `paper_A.download` 与 `paper_B.parse` 可以并发
- `paper_A.parse` 与 `paper_A.refine` 不可并发
- `paper_A.refine` 与 `paper_A.generate_note` 不可并发

### 10.3 Celery 下的并发策略

文献管理模块的异步任务通过 `Celery` + `Redis` 执行，遵循架构概览中的进程模型：

- `FastAPI` 接收请求，创建 `biz_job` 记录，调用 `.delay()` 投递任务到 Redis
- `Celery Worker` 消费并执行任务，Worker 内部通过 `Agno` 调用 LLM
- `Celery Beat` 负责定时任务（如每日论文推送）

SQLite 写锁控制：

- Worker 并发数建议 `--concurrency=4` 以内
- 同一论文的冲突任务由 `resource_key`（`paper:{paper_id}`）在业务层保证串行
- 不同论文的任务可在多个 Worker 间并发执行

这样既能支持多论文同时处理，又能避免 SQLite 写锁冲突。

### 10.4 Checkpoint 机制

人工审查节点是明确的 workflow checkpoint。

调度器应支持：

- 任务进入 `waiting_review`
- 等待用户更新 `refined.md`
- 用户调用确认接口后恢复流程

## 13. 失败与恢复策略

### 13.1 下载失败

处理方式：

- 标记 `download_status = failed`
- 保留论文记录
- 支持重新创建下载任务

### 13.2 解析失败

处理方式：

- 标记对应阶段失败
- 不删除已有 PDF
- 允许重跑 parse / refine

### 13.3 人工审查中断

处理方式：

- 保持 `waiting_review`
- 保留 `parsed/refined.md`
- 用户可稍后继续修改并确认

### 13.4 note 合并冲突

处理方式：

- 标记 `note_status = conflict_pending`
- 不自动覆盖用户内容
- 等待用户手动处理或未来提供冲突解决工具

### 2.3 小 goal 提交 GitHub 规则

后续进入集中实现阶段后，按“小 goal”切分提交，避免长时间堆积未提交变更。

**小 goal 定义**：

- 一个后端端点或一组强相关端点打通。
- 一个前端页面/组件完成真实 API 接入。
- 一个 Worker 任务或一条状态流转完成。
- 一个测试层级补齐并能独立验证。
- 一个明确技术债修复完成，例如 import failure、schema drift、单文件拆分中的一个子模块。

**提交规则**：

- [ ] 每完成一个小 goal，先运行该 goal 的最小验证命令。
- [ ] 验证通过后立即 `git add` 相关文件。
- [ ] 使用 Conventional Commits，例如 `feat(papers): async pipeline dispatch`、`test(worker): cover paper task flow`。
- [ ] 提交后推送到 GitHub 当前工作分支。
- [ ] 在本清单对应条目后补充 commit hash 或 PR 链接，便于追踪。
- [ ] 不把无关目标混进同一个 commit；不在测试未跑或已知失败未说明时提交。

**推荐命令模板**：

```powershell
git status -sb
git add <files-for-this-small-goal>
git commit -m "<type>(<scope>): <summary>"
git push origin <current-branch>
git rev-parse --short HEAD
```

### 2.4 小 goal 完成定义模板

每个小 goal 完成时，在对应条目下补充以下信息，避免“功能写了但无法验收”：

```markdown
完成记录：
- Commit:
- 验证命令:
- 验证结果:
- 已知风险:
- 后续依赖:
```

小 goal 的最低完成定义：

- [ ] 代码或文档改动只覆盖当前 goal，不夹带无关改动。
- [ ] 后端改动至少有 unit/API/import smoke 中的一类验证。
- [ ] 前端改动至少通过 `cmd /c npm run build`，涉及交互时补充截图或浏览器 smoke。
- [ ] API 契约变更同步更新 `docs/03_api/接口规范.md` 或本清单覆盖矩阵。
- [ ] 失败路径有明确错误信息，不只验证 happy path。
- [ ] 提交后回填 commit hash。

### 2.5 第一批建议推进顺序

先做能提升确定性的“小而硬”目标，再进入长链路异步化：

| 顺序 | 小 goal | 原因 | 最小验证 |
|------|--------|------|----------|
| 1 | P0-0 测试/import 地基修复 | 当前 pytest 与缺失 import 会影响所有后续判断 | import smoke + 后端最小 pytest |
| 2 | P1-17 接口覆盖矩阵落表 | 先明确哪些后端能力没有 UI/API client | 文档 diff review |
| 3 | P1-18 API client 拆分 | 为 Jobs/Projects/Resources/Config 接入降复杂度 | `cmd /c npm run build` |
| 4 | P1-19 Jobs API 前端打通 | 通知中心和异步 pipeline 都依赖 Job 查询 | API client smoke + build |
| 5 | P1-25 至 P1-30 任务通知中心 | 让后续所有异步任务有统一用户反馈 | 构造 running/failed/succeeded job + build |

> P0-0 是本清单新增的隐含前置项：修复 pytest 临时目录 ACL、缺失 `runtime.py` import、增加 import smoke。完成后可把 P0-1 到 P0-6 作为异步化主线连续推进。

## 11. 测试方案与验收门禁

### 11.1 测试分层

| 层级 | 测试内容 | 建议覆盖 |
|------|----------|----------|
| Import smoke | 核心模块能否无副作用导入 | `app.main`、`worker.app`、`core.services.papers.knowledge`、`core.services.projects` |
| Unit | 纯函数与局部 runtime | parser/refine/split/note merge/LLM JSON repair/managed block merge |
| Repository | SQLite 与文件系统一致性 | schema 初始化、迁移、asset、artifact、doc version、note_state |
| API | FastAPI 契约 | Paper/Project/Resource/Category/Config/Job 的 CRUD、错误码、分页、冲突检测 |
| Worker | Celery 任务契约 | task name 注册、eager mode 执行、Job 状态流转 |
| Pipeline | 主业务闭环 | Paper 全链路、Project 六模块任务、Knowledge/Dataset 抽取 |
| Frontend-Backend Contract | 前后端接口一致性 | API client 参数、envelope 解析、错误映射、页面入口 |
| Notification UI | 任务通知中心 | Job 轮询、运行中/失败/完成展示、资源跳转 |
| Frontend | 类型与构建 | `tsc -b`、`vite build`、API client 类型对齐 |
| Browser E2E | 用户可操作闭环 | 导入论文、等待 Job、确认 review、打开 PDF/note、进入 Project |
| Real provider | 外部服务真实探测 | gPaper、MinerU、LLM、Zotero MCP；默认 gated |

## 12. 风险登记册

| 风险 | 影响 | 触发信号 | 缓解策略 | 优先级 |
|------|------|----------|----------|--------|
| Windows pytest 临时目录 ACL | 无法判断后端测试真实状态 | `PermissionError`、`pytest-of-*` 或 `.uv-cache` 不可读 | 固定仓库内可写 basetemp；清理坏 ACL；增加 import smoke | P0 |
| 缺失 `knowledge.runtime` | import 链路不稳定，Worker/API 启动可能失败 | `ModuleNotFoundError` | 先补最小 runtime，再迭代真实抽取 | P0 |
| SQLite 并发写入 | Celery 异步化后 Job/Paper 状态可能锁表 | `database is locked`、任务随机失败 | 短事务、重试、单 writer 约束；必要时迁移 Postgres | P0 |
| API envelope 漂移 | 前端错误解析和契约测试不稳定 | 文档 `{code,message,data}` 与实现 `{data,meta,error}` 不一致 | 先定最终 envelope，再统一 docs/API client/tests | P1 |
| Jobs 状态语义不统一 | 通知中心和 pipeline 进度展示混乱 | 同一任务无法区分 queued/running/stage | 统一 JobRecord 字段和状态机，必要时增加 `stage` | P1 |
| LLM/provider 失败 | Knowledge/Dataset/Project 生成不可用 | timeout、quota、JSON parse failure | gated real-provider tests；fallback message；JSON repair + schema validation | P1 |
| 前端静态页与后端能力漂移 | 看起来有页面但没有真实数据闭环 | 页面仍使用 mock data | 接口覆盖矩阵逐项打勾，禁止只做静态 UI 验收 | P1 |
| Celery/Redis 环境缺失 | 异步任务本地无法启动 | Worker 启动失败、broker 连接失败 | 支持 eager mode 测试；文档化本地启动命令；健康检查 | P1 |
| 大文件 `PaperRepository` 继续膨胀 | 后续修改冲突和回归风险上升 | 单文件超过 2500 行，职责混杂 | 按 documents/artifacts/pipeline/note_state 渐进拆分 | P1 |
| 推送粒度失控 | 梭哈阶段难以回滚 | 一个 commit 混多个目标 | 强制小 goal commit，清单回填 commit hash | P1 |

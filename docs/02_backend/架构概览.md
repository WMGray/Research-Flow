# Research-Flow 后端架构概览

## 1. 文档目标

本文档用于定义 Research-Flow 后端的总体技术架构、模块边界、存储策略、任务编排方式与外部服务接入原则，作为后续模块设计、数据库设计、接口细化与代码实现的统一依据。

本文档回答的问题是：

- 后端采用什么技术基线
- 系统如何按层划分
- `Paper`、`Project`、`Knowledge`、`Dataset`、`Presentation` 如何协同
- 文件系统与 SQLite 如何分工
- 异步任务如何支持并发执行与串行依赖
- LLM / Agent 如何通过通用 provider 抽象接入
- 当前首版哪些能力优先实现，哪些顺序后置

本文档不展开：

- 前端页面与交互细节
- 具体数据库字段级设计
- 单个接口的请求响应参数细节
- 单个第三方服务的接入配置说明

---

## 2. 当前架构基线

### 2.1 技术栈

当前后端技术方案采用以下基线：

- Web 框架：`FastAPI`
- Agent / LLM orchestration：`Agno`
- 异步任务队列：`Celery` + `Redis`
- 本地数据库：`SQLite`
- 核心存储：本地文件系统

### 2.2 技术选型原则

- 优先保证单机、本地优先、可快速迭代
- 先打通从论文入库到课题沉淀的完整链路
- 保留对多 provider LLM 的通用抽象，不把业务逻辑绑死在单一平台
- 对未来任务并发、provider 扩展、数据库迁移保留升级空间

### 2.3 当前阶段边界

当前首版以“平台主链路可跑通”为首要目标：

- 除 `Zotero` 外，`Paper`、`Project`、`Knowledge`、`Dataset`、`Presentation`、Daily Feed、Recommendation、会议追踪、投稿/Rebuttal、录用后处理都进入首版实现范围
- 不同能力按 `P0 / P1 / P2` 区分优先级、实现顺序与完成度要求
- `Presentation` 与 Daily Feed 为 P1；Recommendation、会议追踪、投稿/Rebuttal、录用后处理为 P2 收尾补齐能力
- Zotero 集成明确延后

### 2.4 目标与实现边界

当前文档中的多数章节描述的是目标架构和模块设计，不代表所有能力已经在代码中完整落地。实现状态以 `backend/core/`、`backend/app/api/`、测试用例和回归结果为准，临时状态报告不作为长期设计文档进入 `docs`。

截至 2026-05-01，当前代码已经具备 `Paper`、`Project`、`Job`、`Category` 与 `Config / Agent / Skill` 的 P0 本地闭环雏形，但以下能力仍属于部分实现或设计中：

- `Paper` 下载、解析、章节切分、Knowledge / Dataset 抽取还没有完全接入目标中的真实异步 provider 链路。
- `Project` 已支持六模块初始化、文档读写、论文关联、删除、模块级生成任务和跨资源聚合的本地最小实现。
- `Category` 已支持树查询、创建、更新和删除；`Config / Agent / Skill` 已支持本地配置种子、查询与更新，LLM status 当前为配置状态占位，真实首字时延探测待接入。
- `Presentation`、`Dataset`、`Knowledge` 已有本地 CRUD 或占位生成能力；Daily Feed、Recommendation、Conference 当前仍未完整后端 CRUD。
- 当前 SQLite schema 是 P0 精简子集，不等同于 `表设计.md` 的完整目标表结构。

## 3. 系统定位与核心对象

系统围绕五类核心资源组织：

- `Paper`：论文入库、解析、分节、分析与 `note` 生成的核心起点
- `Project`：围绕研究问题沉淀长期材料与六模块工作区
- `Knowledge`：从论文中提炼出的结构化观点、定义、特征
- `Dataset`：论文与实验中涉及的数据集档案
- `Presentation`：组会汇报与阶段性输出资源

资源优先级为：

`Paper > Project > Presentation > Knowledge > Dataset`

设计原则是：

- 资源中心化：所有生成、抽取、分析能力都挂载在资源上
- 本地优先：正文、PDF、Markdown、导出物进入文件系统
- 数据索引化：SQLite 保存元数据、关系、状态、配置与任务
- 流程任务化：下载、解析、优化、生成等都以 `Job` 表示

## 4. 总体分层架构

后端采用分层架构，概念上拆分为五层；代码组织上保持三块清晰边界：

```text
backend/app/      FastAPI 入口：HTTP API、请求响应 Schema、任务投递
backend/worker/   Celery 入口：任务注册、任务执行、定时调度
backend/core/     共享能力：模型、配置、存储、业务逻辑、任务名契约
```

核心约束：

- `app/` 和 `worker/` 都是入口层，不承载可复用业务能力。
- `worker/` 独立部署时可以携带 `core/`，但不能依赖 `app/`。
- `app/tasks/` 与 `worker/tasks/` 文件结构保持一一对称，任务名统一来自 `core/task_names.py`。
- 当前阶段已完成核心共享能力迁移：配置加载、存储路径、资产 helper、schema 片段、Paper 主链路、Project P0、LLM、MCP、论文下载与 PDF 解析已迁入 `core/`；`app/` 只保留 HTTP API、请求/响应 schema、生命周期和任务投递入口。

### 4.1 API 层

职责：

- 暴露 REST API
- 做输入校验、权限边界与响应封装
- 创建任务、查询资源、返回状态

不负责：

- 直接执行业务流程
- 直接调用 provider
- 直接写复杂文件编排逻辑

### 4.2 Application / Service 层

职责：

- 编排核心业务流程
- 决定资源状态流转
- 决定任务依赖与任务触发时机
- 协调数据库、文件系统、provider 与任务系统

目标代码位置为 `core/services/`。新增共享业务优先直接落在 `core/services/`；当前已经迁入 `papers`、`projects`、`llm`、`mcp`、`paper_download` 与 `pdf_parser`。API 层负责 Pydantic schema 与 core DTO 之间的转换。

典型服务：

- `PaperService`
- `ProjectService`
- `KnowledgeExtractionService`
- `DatasetService`
- `PresentationService`
- `JobService`
- `ConfigService`

### 4.3 Domain / Workflow 层

职责：

- 表达核心业务规则与流程约束
- 定义单个资源的状态机
- 定义任务依赖、人工审查、增量更新策略

典型对象：

- `PaperWorkflow`
- `ProjectWorkflow`
- `JobDependencyPolicy`
- `NoteMergePolicy`
- `ReviewCheckpointPolicy`

### 4.4 Repository / Storage 层

职责：

- 封装对 `SQLite` 的访问
- 封装对本地文件系统的访问
- 保证数据库记录与文件路径映射一致

目标代码位置为 `core/models/`、`core/database.py` 与 `core/storage.py`，因为 FastAPI 与 Celery Worker 都需要访问这些能力。

典型组件：

- `PaperRepository`
- `ProjectRepository`
- `JobRepository`
- `AssetRepository`
- `FileStorage`

`SQLite` 部分采用与表设计一致的四段式思路：

- 物理层：记录真实文件与目录位置
- 资产层：为文件、目录与逻辑对象提供统一资产 ID
- 关系层：显式记录资源之间的关联关系
- 业务层：承载 `Paper`、`Project`、`Knowledge`、`Dataset`、`Presentation` 的领域字段

这样做的好处是：

- 文件系统与数据库可解耦演进
- 同一份物理文件可以被逻辑资源引用
- 后续知识图谱、导出物追溯、任务产物归属都有统一挂载点

### 4.5 Integration 层

职责：

- 对接 PDF parser、LLM provider、Agno agent、云服务与未来的 Zotero
- 将不同 provider 统一成稳定协议

典型组件：

- `PdfParserAdapter`
- `LlmProvider`
- `AgentRunner`
- `ProviderRegistry`

PDF 解析与 LLM 优化解析的细化设计见 [services/pdf-parser.md](./services/pdf-parser.md)。该链路以 MinerU `full.md` 为主输入，通过 `diagnose -> repair patch -> apply -> verify` 生成 `parsed/refined.md`，避免让 LLM 直接自由重写整篇论文。

## 5. 目录与存储布局

### 5.1 总体原则

- 文件系统保存重内容与产物
- SQLite 保存轻量索引、关系、状态与配置
- 每个资源都必须能同时在数据库中被索引、在文件系统中被追溯
- 资源目录统一采用 slug 化命名，展示名与目录名分离

### 5.1.1 slug 规则

为了保证跨平台可移植性与路径稳定性，`PaperName`、`ProjectName`、分类路径都采用 slug 化目录名。

建议：

- 数据库存展示原始标题和原始名称
- 文件系统使用 `direction_slug`、`sub_direction_slug`、`paper_slug`、`project_slug`
- slug 生成后尽量稳定，不随轻微标题调整频繁变化
- 需要保留重命名能力，但应通过迁移逻辑统一更新数据库映射

### 5.2 Paper 目录布局

论文统一落在 `Papers/` 顶层目录下，按“大方向 / 小方向 / PaperName” 组织，落盘时使用 slug 化路径。

建议目录结构：

```text
Papers/
  {direction_slug}/
    {sub_direction_slug}/
      {paper_slug}/
        paper.pdf
        note.md
        metadata.json
        parsed/
          raw.md
          refined.md
          sections/
            01_intro.md
            02_method.md
            ...
        extracted/
          knowledge.json
          datasets.json
        jobs/
          history.json
```

说明：

- `metadata.json` 为文件侧缓存，不替代数据库
- 人工审查直接复用 `parsed/refined.md`，不额外创建 `review.md`
- `note.md` 是统一研究笔记，不拆出独立的 human note

### 5.3 Project 目录布局

课题统一落在 `Projects/` 顶层目录下，按 `ProjectName` 组织，落盘时使用 slug 化路径。

建议目录结构：

```text
Projects/
  {project_slug}/
    overview.md
    related-work.md
    method.md
    experiment.md
    conclusion.md
    manuscript.md
    assets/
    exports/
    jobs/
```

### 5.4 导出物布局

导出物进入各自资源目录，不集中放到单独的全局导出目录。

例如：

- 论文相关导出物放在对应 `Papers/.../{PaperName}/`
- 课题相关导出物放在 `Projects/{ProjectName}/exports/`
- 未来汇报导出物放在对应 `Presentation` 所属目录

### 5.5 数据库存储内容

SQLite 主要保存：

- 资源主键与元数据
- 文件逻辑路径与状态
- 资源之间的显式关系
- 异步任务状态与依赖
- LLM / Agent 配置
- Skill runtime instruction
- 探活与观测结果

## 6. 核心业务链路

### 6.1 Paper 主链路

`Paper` 是当前最优先落地的主链路：

1. 用户通过 DOI、URL 或标题创建论文记录
2. 系统创建 `download` 任务，获取或挂接 `paper.pdf`
3. 系统创建 `parse` 任务，调用 MinerU 生成 `parsed/raw.md`，其中 `raw.md` 对应 MinerU `full.md`
4. 系统创建 `refine` 任务，对 `raw.md` 执行 LLM 结构诊断、局部 patch 修复与 verifier 检查，生成 `parsed/refined.md` 和 `parsed/refine/*.json`
5. 系统进入人工审查节点，等待用户修改并确认 `refined.md` 与低置信度 review items
6. 用户确认后，系统创建 `split` 任务，基于确认后的 `refined.md` 生成章节结果
7. 系统创建 `analyze` / `note_generate` 任务
8. 系统更新 `note.md`
9. 系统触发 `Knowledge` 与 `Dataset` 抽取

### 6.2 Project 链路

`Project` 在 `Paper` 主链路稳定后接入：

1. 用户创建课题
2. 系统初始化六个 Markdown 模块
3. 用户关联论文并标注关系类型
4. 用户手动触发 related work、method、experiment 等模块任务
5. 生成结果写回对应模块文件

### 6.3 Knowledge / Dataset 链路

`Knowledge` 与 `Dataset` 在当前阶段以“从论文链路中抽取并落库”为主：

- `Knowledge`：抽取观点、定义、特征及证据片段
- `Dataset`：抽取数据集名称、任务类型、评价方式、使用论文

当前阶段不做复杂工作台，只保证：

- 可稳定抽取
- 可稳定落库
- 可与论文、课题建立关系

`Knowledge` 的首版字段应尽量完整，至少覆盖：

- 中文总结
- 英文原文
- 知识类型，如定义 / 观点 / 特征
- 所属类别或研究方向
- 来源论文与证据位置
- 抽取置信度
- 后续人工修订状态

### 6.4 Presentation 链路

`Presentation` 在当前首版中需要实现，但优先级低于 `Paper` 与 `Project`：

- 允许从 `Project` 创建 Presentation 资源
- 允许记录输入材料与导出目标
- 允许生成中间草稿与导出结果
- 当前阶段可先实现较简化但完整可用的生成链路

## 7. 任务系统设计

### 7.1 设计目标

任务系统需要同时支持：

- 多个同类型任务并发执行
- 多个不同类型任务并发执行
- 同一资源上的任务按依赖顺序串行推进
- 人工审查节点暂停与恢复

典型场景：

- 多篇论文同时下载 PDF
- 一篇论文在下载，另一篇论文在解析
- 多篇论文同时进行不同阶段处理
- 同一篇论文不能同时执行冲突步骤

### 7.2 任务模型

建议将任务拆成三个概念：

- `JobDefinition`：任务类型定义，例如 `download_pdf`、`parse_pdf`
- `JobInstance`：某次具体执行，例如“对 paper_A 执行下载”
- `WorkflowCheckpoint`：流程中的暂停点，例如人工审查待确认

### 7.3 并发原则

并发控制建议遵循以下规则：

- 不同资源上的任务允许并发
- 同类型任务在不同资源上允许并发
- 不同类型任务在不同资源上允许并发
- 同一资源上的冲突任务必须串行

示例：

- `paper_A.download_pdf` 与 `paper_B.download_pdf` 可以并发
- `paper_A.download_pdf` 与 `paper_B.parse_pdf` 可以并发
- `paper_A.parse_pdf` 与 `paper_A.refine_parse` 不能无依赖并发
- `paper_A.refine_parse` 与 `paper_A.note_generate` 不能直接并发

### 7.4 资源级锁

为了实现“跨资源并发、单资源有序”，任务系统应引入资源级锁或资源级串行队列。

建议引入：

- `resource_key`：如 `paper:{paper_id}`
- `job_stage`：如 `download`、`parse`、`refine`、`review`、`split`
- `dependency_job_id` 或阶段依赖规则

约束：

- 同一 `resource_key` 上只允许一个互斥阶段任务处于 `running`
- 非互斥的只读任务未来可放宽
- 当前阶段先采用保守策略，即同一论文主链路任务单线程推进

### 7.5 Celery + Redis 任务协调策略

当前阶段采用 `Celery` + `Redis` 作为异步任务基础设施，进程架构如下：

```
FastAPI (端口 8000)  ──投递──▶  Redis Broker  ──消费──▶  Celery Worker (可横向扩展)
                                                          ▲
                                              ┌─────────┘
                                   Cron 定时投递
                                   Celery Beat (单实例)
```

- **FastAPI**：接收用户请求，通过 `app/tasks/` 投递异步任务；任务名来自 `core/task_names.py`，不直接执行耗时操作
- **Celery Beat**：按 `worker/schedules.py` 定义 Cron 表达式定时投递周期性任务（如每日论文推送），全局仅一个实例
- **Celery Worker**：消费 Redis 队列中的任务并执行，可按需水平扩展；Worker 任务入口调用 `core/services/` 中的业务能力，并在需要时通过 `Agno` 完成 LLM/Agent 编排

当前代码边界为：通用服务进入 `core/services/`，FastAPI 入口只做请求/响应转换与错误映射；新的 Worker 任务不应直接依赖 `app/`。

#### SQLite 并发控制

由于 `SQLite` 写锁限制，当前推荐策略：

- `FastAPI` 负责数据库读操作和创建任务记录
- Celery Worker 内的写操作通过资源级锁控制并发
- 同一资源（如论文）上的互斥任务串行推进，不同资源之间允许并发

首版建议：

- Worker 并发数有限控制（推荐 `--concurrency=4` 以内）
- 同一 `resource_key` 的互斥阶段任务由业务层保证不并发
- 数据库写操作在 Worker 内串行化到同一连接

后续若迁移到 `PostgreSQL`，可以放开 Worker 数量和并发限制。

### 7.6 任务状态机

建议统一状态：

- `queued`
- `running`
- `waiting_review`
- `waiting_confirm`
- `succeeded`
- `failed`
- `cancelled`

其中：

- `waiting_review`：系统已生成待审查文档，等待用户修改
- `waiting_confirm`：用户修改完成但尚未确认继续

## 8. 人工审查与确认机制

### 8.1 节点位置

人工审查节点位于：

`LLM 优化解析` 之后，`章节切分` 之前

### 8.2 工作方式

当前方案不是“纯暂停不产物”，而是：

1. 系统完成 `refined.md`
2. 用户直接在 `parsed/refined.md` 上进行人工修改
3. 任务状态进入 `waiting_review`
4. 用户完成修改后点击确认
5. 系统创建或恢复后续任务，进入 `split`

### 8.3 与“自动继续”的区别

自动继续模式的特点是：

- 无人工检查
- `refined.md` 直接进入后续切分
- 速度快，但不可控

人工审查确认模式的特点是：

- 用户会修改中间文档
- 后续步骤必须以确认后的版本为输入
- 更适合首版对解析质量进行兜底

因此当前架构应显式支持“审查暂停 -> 用户修改 -> 用户确认 -> 恢复流程”。

## 9. note 增量更新策略

### 9.1 单文档原则

论文级研究笔记统一为单个 `note.md`，系统生成与人工修改都发生在同一文档中。

### 9.2 重生成策略

当前确定采用“合并式增量更新”，不采用无提示覆盖。

目标是：

- 保留用户已经修改的内容
- 仅更新与新分析结果强相关的区块
- 记录最近一次自动生成与人工修改状态
- 能够准确识别当前 `note.md` 是否已被用户修改、是否已合并、是否存在冲突待处理

### 9.3 建议机制

首版建议采用“区块化合并”策略：

- 为系统生成区块设置稳定标记
- 人工自由补充区块默认不自动覆盖
- 再次生成时只更新系统托管区块

例如：

- `Research Question`
- `Core Method`
- `Main Contributions`
- `Experiment Summary`
- `Limitations`

每个区块可保留：

- `generated_at`
- `source_sections`
- `last_merged_at`

建议同时维护 `note_status` 与内容指纹：

- `clean_generated`：纯系统生成，用户未修改
- `user_modified`：检测到用户修改
- `merged`：系统已完成一次增量合并
- `conflict_pending`：存在无法自动安全合并的区块

识别方式建议结合：

- 文档整体 hash
- 系统托管区块 hash
- 最近一次自动生成时间
- 最近一次用户确认时间

这样既能增量更新，也能避免整篇笔记被完全重写。

## 10. LLM / Agent 架构

### 10.1 总体原则

- 保留通用 LLM 抽象
- provider 通过协议接入
- `Agno` 作为统一的 LLM / Agent 管理框架
- 业务流程只依赖统一能力接口，不依赖具体平台 SDK
- 高层任务场景抽象为 Agno scene / capability，并通过 `Agno + provider + skill runtime instructions + tools` 执行

### 10.2 provider 协议

建议定义统一协议：

```python
class LlmProvider(Protocol):
    def chat(self, request: ChatRequest) -> ChatResponse:
        ...

    def stream(self, request: ChatRequest) -> Iterator[ChatDelta]:
        ...

    def probe(self, request: ProbeRequest) -> ProbeResult:
        ...
```

建议 provider registry 根据配置动态装配：

- `openai`
- `deepseek`
- `siliconflow`
- 其他兼容 provider

### 10.3 Agno 运行时编排

`Agno` 是后端执行 LLM / Agent 任务的运行时编排层；Paper 能力的模型指令、输出契约和评估规则统一沉淀在仓库根目录 `skills/` 中。

Agno 在后端中的职责：

- 绑定业务 scene 与 `AgentProfile`。
- 选择 provider、model、temperature、timeout 与 fallback 策略。
- 读取并渲染 Paper skill runtime instructions。
- 组织工具调用，例如文件读取、结构化解析、检索或未来的 Zotero 工具。
- 记录 LLM run、输入摘要、输出摘要、错误与重试信息。
- 将结构化输出交回业务 service，由后端代码负责校验、持久化和 artifact 登记。

Agno 不负责：

- 替代业务状态机。
- 直接写数据库最终状态。
- 直接把不可信 LLM 输出写入 `note.md`、`parsed/refined.md` 或结构化表。
- 把 `SKILL.md` 当作业务代码执行；`SKILL.md` 是能力说明，runtime instructions 放在 `references/`。

Celery 与 Agno 的分工：

| 关注点 | Celery / FastAPI | Agno |
|--------|------------------|------|
| HTTP 请求处理 | 负责 | 不参与 |
| 任务投递与队列 | 负责 | 不参与 |
| 定时调度与重试 | 负责 | 不参与 |
| 业务状态机推进 | core service 负责 | 不直接负责 |
| provider / model 调度 | 不直接负责 | 负责 |
| Agent profile 选择 | 提供配置入口 | 负责执行时解析 |
| skill runtime instruction 渲染 | 提供 instruction key 与变量 | 负责或协助执行 |
| 工具调用编排 | 提供业务工具边界 | 负责调用与聚合 |
| 输出校验与落库 | validator / repository 负责 | 只返回结构化结果 |

当前阶段不做复杂多 agent 编排中心，只做 scene 级 agent 配置、provider / model 路由、工具集挂载和可替换的 agent runtime。Knowledge / Dataset 抽取、章节切分、note 生成等 Paper LLM 能力都应表达为 Paper skill，而不是单独的 prompt 文件或额外配置层。

推荐执行链路：

```text
FastAPI / Celery
  -> core service
  -> job record
  -> Agno scene runner
  -> LLM provider / tools
  -> structured result
  -> validator / renderer / repository
  -> artifact + DB state
```

### 10.4 Skill 目录约定

仓库根目录 `skills/` 是 Paper LLM 能力的唯一沉淀位置。每个 skill 同时服务两类读者：

- Codex / 开发 agent：读取 `SKILL.md` 理解能力边界、修改规则和验证方法。
- 后端运行时：通过 `backend/core/services/papers/skill_runtime.py` 读取 `references/runtime-instructions.md` 等模型指令文件。

通用目录结构采用官方式 skill 包：

```text
skills/
  {skill-name}/
    SKILL.md              # 必需：frontmatter + 精简工作流说明
    agents/
      openai.yaml         # 推荐：UI 展示名、短描述、默认入口语
    references/           # 可选：契约、schema、rubric、案例
    scripts/              # 可选：确定性工具脚本
    assets/               # 可选：模板、样例资产
```

约束：

- `SKILL.md` 必须包含 YAML frontmatter，至少有 `name` 与 `description`。
- `SKILL.md` 只保留核心流程、边界和验证方法；详细 schema、例子、rubric 放入 `references/`。
- 不在 skill 目录里放 `README.md`、安装指南、变更日志等辅助文档。
- 不引入 `skill.toml` 作为 skill 标准结构。
- 不在 `backend/config/prompts/` 维护 Paper standalone prompts。Paper LLM 指令统一放在对应 skill 的 `references/` 下。

### 10.5 Skill 命名与读取边界

为了避免混淆，本文档统一区分：

| 名称 | 用途 | 位置 | 读取者 |
|------|------|------|--------|
| Paper skill | Paper LLM 能力包，包含说明、契约和 runtime instructions | `skills/{skill-name}/` | Codex / 后端 runtime |
| Agno scene / capability | 后端运行时的一次 LLM 能力调用 | core service / future config table | 后端 service / Agno |
| skill runtime instruction | 后端渲染的模型指令入口 | `skills/{skill-name}/references/*.md` | `skill_runtime.py` |
| feature routing | scene 到模型的路由 | `backend/config/settings.toml` 或未来配置表 | `llm_registry` |

Instruction key 使用历史兼容的 snake_case（如 `paper_refine_parse.diagnose`），skill 目录使用官方 kebab-case（如 `paper-refine-parse/`）。`skill_runtime.py` 按约定将 snake_case skill key 转换为 kebab-case skill 目录，并只保留少量历史别名；不再维护 TOML 或数据库式模型指令映射层。

### 10.6 Paper Skill 分级

Paper LLM 能力按链路层级分组：

| 分级 | Skill | 目录 | 角色 |
|------|-------|------|------|
| P0 Orchestrator | `paper-pipeline` | `skills/paper-pipeline/` | 维护 Paper 端到端工程链路 |
| P0 Core | `paper-refine-parse` | `skills/paper-refine-parse/` | MinerU Markdown refine、patch contract 与验证 |
| P0 Core | `paper-sectioning` | `skills/paper-sectioning/` | canonical section line-range 规划 |
| P0 Core | `paper-note-generate` | `skills/paper-note-generate/` | 深度论文笔记、图表证据与 managed blocks |
| P1 Extraction | `paper-knowledge-mining` | `skills/paper-knowledge-mining/` | 抽取观点、定义、特征、原文证据与引用标记 |
| P1 Extraction | `paper-dataset-mining` | `skills/paper-dataset-mining/` | 抽取 dataset、benchmark、任务、规模、来源和指标 |
| P2 Review | `paper-figure-table-reading` | 待建 | 图表角色判断与解释规则 |
| P2 Review | `paper-evidence-checking` | 待建 | 审查笔记或抽取结果是否有证据、是否编造 |

这些 skills 可以独立迭代，但 Paper runtime 不再维护独立 prompt 文件。

### 10.7 后端运行时配置

配置按“处理环节”组织，而不是按页面组织。

至少覆盖：

- 论文优化解析
- 分节后分析
- `note` 生成
- Knowledge 抽取
- Dataset 抽取
- related work 整理
- method 草稿生成
- experiment 建议
- probe 探活

当前落地方式：

- `skills/{paper-skill}/SKILL.md` 保存能力边界、流程和验证方法。
- `skills/{paper-skill}/references/runtime-instructions.md` 保存后端可渲染模型指令。
- `skills/paper-note-generate/references/block-runtime-instructions.md` 保存单 block note 生成指令。
- `backend/core/services/papers/skill_runtime.py` 按目录约定解析 instruction key 到 skill reference。
- `backend/core/services/papers/refine/runtime.py` 暂存 refine 的多阶段 binding。
- `backend/config/settings.toml` 保存 provider、model 与 feature 路由。

目标态：

- 非敏感默认配置仍可保留在 `settings.toml`。
- provider、model、feature routing 可进入配置表和管理 API。
- `skills/` 继续作为 Paper 能力的源码级 canonical source；如需线上版本化、启停和灰度，应围绕 skill 包版本做管理，而不是复制出一套模型指令表。

### 10.8 兜底策略

每个关键环节配置：

- 主 provider / model
- 备用 provider / model
- 超时与重试策略

运行时优先使用主模型，失败时根据策略切换备用模型。

- 从论文分析结果中抽取结构化内容

- 维护任务队列

职责：

- 当前阶段需要进入首版实现范围

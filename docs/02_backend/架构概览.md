# Backend 架构概览

## 1. 目标

后端参考 `origin/master` 的分层结构，但第一阶段只落地 Paper 全流程。

本轮不引入：

- Celery / Redis
- worker
- skills 自动调用
- LLM Provider 管理
- Project / Dataset / Knowledge 模块

本轮必须落地：

- 根 `data/` 文件系统真源
- JSON 暂存状态与索引
- Paper ingest / migrate / scan / note-template
- 程序式 PDF parser
- 人工确认节点与失败原因记录

## 2. 当前分层

```text
backend/
├─ app/
│  ├─ api/
│  ├─ schemas/
│  ├─ dependencies.py
│  └─ main.py
├─ core/
│  ├─ config.py
│  └─ services/
│     └─ papers/
├─ scripts/
└─ tests/
```

## 3. 分层职责

### 3.1 `app/`

- FastAPI 入口
- router 注册
- request / response schema
- 依赖注入

### 3.2 `core/`

- 共享配置
- 文件系统文献库逻辑
- Paper repository / service / models
- JSON 状态读写
- PDF parser 编排

### 3.3 `scripts/`

- 面向本地库维护的 CLI
- 与 API 共用 `core` 层逻辑
- 支持批处理与回归验证

## 4. 存储原则

- 仓库根 `data/` 是唯一真源
- `backend/` 不再维护独立数据根
- 产物直接落在 `data/Discover`、`data/Acquire`、`data/Library`、`data/templates`
- 当前数据根写死为仓库根 `data/`
- 软件完成后再通过 Config 支持用户指定外部数据根

## 5. JSON 暂存原则

暂不引入数据库。需要持久化的状态写入 JSON：

- batch 状态
- acquire 队列状态
- paper metadata
- parser run 状态
- 失败原因和人工待办

JSON 文件是可恢复状态，不是临时日志。运行日志和一次性调试输出不进入长期状态文件。

## 6. PDF parser 原则

- parser 是 Paper 全流程的一部分，本轮需要落地。
- parser 只做程序式解析，不依赖 LLM Provider。
- parser 输出应包含文本、结构化片段、图片索引和错误信息。
- parser 失败不能中断整个库扫描，应将单篇论文标记为 `parse-failed` 或 `needs-review`。

## 7. 兼容策略

为避免迁移期间打断旧调用，当前仍保留：

- `backend/app/library.py`
- `backend/app/settings.py`

这两个文件只作为兼容包装层使用，真实实现已经下沉到 `core/`。

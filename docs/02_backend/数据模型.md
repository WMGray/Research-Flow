# Backend 数据模型

## 1. 总体原则

第一阶段不引入数据库。所有状态来自文件系统扫描和 JSON 状态文件，可由 CLI / API 重复读取和重建。

数据根：

```text
data/
├─ Discover/
├─ Acquire/
├─ Library/
└─ templates/
```

## 2. PaperRecord

当前 Paper 全流程的核心记录来自文件系统目录扫描和 JSON 状态文件，统一映射为 `PaperRecord`。

- `title`
- `slug`
- `stage`
- `status`
- `domain`
- `area`
- `topic`
- `year`
- `venue`

- `images_path`
- `metadata_path`
- `parser_status`
- `parser_outputs`
- `error`
- `updated_at`

## 3. BatchRecord

## 4. ParserRunRecord

PDF parser 每次运行写入 `ParserRunRecord`。

关键字段：

- `run_id`
- `paper_id`
- `status`
- `source_pdf`
- `text_path`
- `sections_path`
- `images_dir`
- `started_at`
- `finished_at`
- `error`

状态：

- `pending`
- `running`
- `succeeded`
- `failed`
- `needs-review`

## 5. JSON 状态文件

建议按单篇论文存放：

```text
data/Library/<Domain>/<Area>/<Topic>/<slug>/
├─ metadata.json
├─ parser_runs.json
├─ paper.pdf
├─ note.md
├─ parsed/
│  ├─ text.md
│  ├─ sections.json
│  └─ images/
└─ logs/
```

约定：

- `metadata.json` 存论文长期状态。
- `parser_runs.json` 存解析运行历史和最后一次结果。
- `parsed/` 存 parser 可复用产物。
- `logs/` 只存单篇论文相关排错信息，不作为前端主要数据源。

## 7. 状态模型

当前 HomePage 和后端脚本关注以下状态：

- `processed`
- `needs-pdf`
- `needs-review`
- `parse-pending`
- `parse-failed`
- `failed`
- `curated`

状态含义：

- `curated`：已进入 Acquire 队列，尚未正式入库。
- `needs-pdf`：缺少可解析 PDF。
- `parse-pending`：PDF 已存在，等待解析。
- `parse-failed`：PDF 解析失败，需要人工确认或重跑。
- `needs-review`：目录冲突、元数据异常或 parser 产物不可信。
- `processed`：已入库且关键产物可用。
- `failed`：流程性失败，需查看错误原因。

# 后端全局风险排查

## 1. 排查目标

本文档从当前代码实际状态出发，对 Research-Flow 后端每个已实现、部分实现和设计中功能进行问题预判。重点不是重复目标设计，而是回答：

- 当前功能在真实使用时可能在哪里出问题
- 问题会影响哪个资源或用户流程
- 应该先用什么检查方式确认
- 后续修复应按什么优先级推进

排查时间：2026-04-26

排查范围：

- FastAPI 应用入口与路由
- Paper 主链路
- Project 工作区
- Job 记录与异步任务契约
- SQLite / asset / filesystem 存储
- paper-download / PDF parser / LLM / MCP 集成
- Worker / Celery 契约
- 文档、测试与配置一致性

## 2. 优先级定义

| 优先级 | 含义 | 处理策略 |
|--------|------|----------|
| P0 | 可能造成数据覆盖、流程误导、生产不可用或安全风险 | 进入最近修复批次 |
| P1 | 会造成局部功能失败、状态不一致、用户困惑或后续扩展成本升高 | 与对应功能开发一起修 |
| P2 | 当前可接受，但会影响可维护性、文档可信度或未来迁移 | 纳入 roadmap 或重构清单 |

## 3. 全局结论

当前后端更接近“本地 P0 demo + 部分真实 provider 能力”的状态，而不是完整异步生产链路。主要风险集中在四类：

1. **状态机不够硬**：多个动作接口没有严格校验前置状态，用户可以跳过下载、解析、人工审查或章节切分直接进入后续步骤。
2. **数据保护不足**：`note.md` 与 Project 文档允许无 `base_version` 更新；`generate-note` 当前会无条件覆盖已有 note，存在覆盖人工内容的风险。
3. **目标链路与实际链路混合**：Paper 下载、PDF 解析、Knowledge / Dataset 抽取、Job 异步队列在文档中是完整链路，但代码里部分还是占位或同步执行。
4. **持久化模型是精简子集**：当前 schema 没有迁移机制、资源锁、完整关系元数据、配置表、Knowledge / Dataset / Presentation 表，后续补功能时容易出现兼容问题。

建议先修 P0：

- 为 Paper 动作补前置状态校验和幂等策略。
- 防止 `generate-note` 覆盖 `user_modified` 或 `conflict_pending` 的 note。
- 明确 `download_pdf` 字段行为：要么真正触发下载，要么从 request schema 移除。
- 将 `/api/v1/papers/{paper_id}/download` 接入真实下载，或在 API 返回里明确它只是 placeholder。
- 统一 API envelope 与错误格式，避免前端按错误契约集成后出现不一致。

## 4. 功能逐项排查

### 4.1 应用入口与通用路由

| 功能 | 可能问题 | 影响 | 优先级 | 建议检查 / 修复 |
|------|----------|------|--------|-----------------|
| `/health` | 只返回进程存活，不检查 SQLite、配置、LLM、MCP | 健康检查可能误报可用 | P1 | 增加 `/health/deep`，检查 DB 可写、配置可加载、关键目录可写 |
| 路由挂载 | `/paper-download/*` 没有 `/api/v1` 前缀，与主 API 风格不同 | 前端和文档集成容易混淆 | P1 | 明确 legacy 标记，或迁移到 `/api/v1/paper-download/*` |
| API envelope | 当前实现为 `data/meta/error`，错误多为 FastAPI `detail`，与目标规范不同 | 前端错误处理不稳定 | P0 | 统一 envelope 和 exception handler |
| FastAPI lifecycle | `zotero_client.close()` 只有应用退出时执行，单请求失败不会重置连接 | MCP 子进程连接失败后可能保持坏状态 | P1 | MCP 连接失败时清空 toolkit，并提供 reconnect 逻辑 |

### 4.2 配置加载

| 功能 | 可能问题 | 影响 | 优先级 | 建议检查 / 修复 |
|------|----------|------|--------|-----------------|

### 4.3 SQLite / Asset / 文件系统

| 功能 | 可能问题 | 影响 | 优先级 | 建议检查 / 修复 |
|------|----------|------|--------|-----------------|
| schema 初始化 | 每个 repository 初始化都 `executescript`，没有迁移版本 | schema 演进时无法安全升级已有库 | P0 | 增加 schema version / migration 表 |
| SQLite 并发 | 没有 WAL、busy_timeout、资源级锁 | 多 worker 或并发写入时 `database is locked` | P0 | 配置 WAL + busy_timeout；同资源写操作加锁 |

| 软删除 | Paper 软删除后 asset_link 仍保留 | 长期积累无效关系 | P2 | 删除资源时标记或清理相关关系 |

2026-04-26 当前处理：
- `generate-note` 已要求存在 canonical sections，否则返回 `PAPER_SECTIONS_MISSING` failed job。
- `note_status=user_modified/merged` 时执行 RF managed-block 合并，保留用户手写内容；`conflict_pending` 时拒绝覆盖。
- `download_pdf=true` 已在创建 Paper 后触发 `/download` 逻辑，并返回 `download_job_id`。

| 测试覆盖 | 当前 API happy path 较多，状态非法流转覆盖不足 | 数据保护问题不容易暴露 | P0 | 增加状态机负向测试 |
| fixture 隔离 | 测试设置 env 后部分没有 `reset_settings()` | 配置缓存可能污染其他测试 | P1 | 所有配置相关 fixture 做 teardown reset |
| pytest 临时目录 | 当前环境下 pytest 创建的 basetemp 目录会触发 `PermissionError: Access denied` | 本地回归测试无法稳定跑完，且会留下不可访问临时目录 | P1 | 固化可访问的测试临时目录策略，或清理现有受限目录权限 |
| 文档链接 | 中文文件路径在部分工具中可能编码显示异常 | 审阅体验受影响 | P2 | 保持相对链接，避免绝对 Windows 路径 |

## 5. P0 排查清单

建议优先补这些测试，再动实现：

1. 创建 Paper 时 `download_pdf=true` 的行为必须被测试覆盖。
2. 未下载 PDF 时调用 `/parse` 应失败，而不是生成占位 raw markdown。
3. 未 refine 成功时 `submit-review` 应失败。
4. 未进入 `waiting_review` 时 `confirm-review` 应失败。
5. 未确认 review 时 `split-sections` 应失败。
6. `note_status=user_modified` 时 `generate-note` 不得覆盖用户内容。
7. PUT note / Project document 不传 `base_version` 时应拒绝，或必须显式 `force=true`。
8. `extract-knowledge` / `extract-datasets` 在真实抽取未实现前不应返回 `succeeded`。
9. Celery `beat_schedule` 中的任务必须有 worker 注册实现，或测试应标记 schedule disabled。
10. API 错误响应格式应统一，前端只需要处理一种错误 envelope。

## 6. 修复顺序建议

### 第一批：保护数据与状态

- Paper 动作前置条件校验。

- API envelope 与错误 handler 统一。

- Category CRUD。

- Presentation / Feed / Conference 后置实现。

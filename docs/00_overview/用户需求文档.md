# Research-Flow 用户需求文档

## 1. 文档说明

### 1.1 编写目的

本文档用于完整、清晰地描述 Research-Flow 平台的用户需求，作为后续系统设计、功能开发与验收测试的依据。

### 1.2 适用范围

本文档适用于 Research-Flow 项目的产品设计、后端实现、自动化能力设计与最终验收。

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| 文献入库 | 将论文的 PDF、元数据、解析结果与研究笔记统一纳入平台管理 |
| 解析源文档 | PDF 解析工具直接输出的原始 Markdown 或结构化文本 |
| 优化解析文档 | 基于 LLM 对解析源文档进行清洗、纠错与结构修复后的 Markdown 文档 |
| note | 基于论文解析结果生成的研究笔记文档，允许用户在同一文档中继续修改，系统记录生成状态、人工编辑状态与最后更新时间 |
| Knowledge | 由观点、定义、特征等结构化知识条目组成的资源集合 |
| 课题 | 用户正在研究或已归档的具体研究项目 |
| Presentation | 课题阶段性汇报对应的演示材料资源，包括大纲、逐页草稿与导出文件 |
| Agent | 面向特定任务场景配置的自动化执行单元，可绑定模型、提示词与任务策略 |
| 知识图谱 | Paper、Dataset、Project、Presentation之间关系的结构化图状表示   |
| Rebuttal | 针对审稿意见的书面回复内容 |

## 2. 项目背景与目标

### 2.1 背景

科研人员在研究过程中普遍面临以下问题：

- 文献数量庞大，手动整理效率低，来源分散，容易遗漏。
- 论文解析质量不稳定，原始 PDF 解析结果存在格式错乱、章节识别不完整等问题。
- 研究笔记缺乏统一结构，难以横向比较、沉淀和复用。
- 课题推进信息分散在多份文档中，缺乏统一视图。
- 汇报材料准备高度依赖手工整理，重复劳动较多。
- 论文写作、实验规划与知识提炼之间缺乏统一支撑平台。
- 学术会议的重要时间节点分散，容易错过关键截止日期。

### 2.2 目标

Research-Flow 定位为面向计算机方向 PhD 与科研工作者的研究管理平台，用于统一管理论文、课题、汇报材料及相关知识资产。

平台边界如下：

- 平台仅面向科研管理，不承担软件开发代码管理、工程开发协作与代码托管职责。
- 平台重点覆盖文献管理、课题管理、知识沉淀、实验规划、汇报准备与论文写作辅助。
- 平台采用本地优先策略，首阶段基于本地文件系统与本地数据库运行，不引入 OSS 等远程对象存储。

资源组织原则如下：

- 核心资源为 `Paper` 与 `Project`。
- `Presentation`、`Knowledge`、`Dataset` 为围绕核心资源衍生出的辅助资源。
- 当前资源优先级为：`Paper > Project > Presentation > Knowledge > Dataset`。

### 2.3 典型工作路径

平台以资源管理为主线，典型使用路径如下：

1. 论文路径
   用户通过 DOI、URL 或标题导入论文，指定分类后由系统完成下载、PDF 解析、LLM 优化解析、章节切分、note 生成与入库。
2. 课题路径
   用户创建课题并维护六个核心模块，按需关联论文、实验记录与知识条目，再手动触发 LLM 分析或总结任务。
3. 组会汇报路径
   用户创建汇报任务，选择当次汇报所需的论文、课题模块与实验材料，系统生成可编辑的汇报大纲和逐页草稿。

## 3. 用户角色

### 3.1 主要用户：计算机方向 PhD 与科研工作者

- 主要面向计算机方向博士生、研究人员与科研助理。
- 需要管理大量论文、维护多个课题、定期准备汇报材料。
- 需要在研究过程中持续积累结构化知识、实验规划依据与论文写作素材。

### 3.2 用户核心诉求

| 诉求 | 说明 |
|------|------|
| 高效入库 | 支持通过 DOI、URL、标题等方式快速导入论文，并自动完成后续处理链路 |
| 稳定解析 | 在原始 PDF 解析基础上引入 LLM 优化步骤，提高章节识别与内容可读性 |
| 统一笔记 | 每篇论文形成统一的 `note` 文档，既可自动生成，也可由用户直接继续修改 |
| 课题沉淀 | 将相关论文、实验规划、阶段结论与写作草稿集中沉淀到课题空间 |
| 汇报复用 | 从已积累的论文与课题材料中快速组织组会汇报内容 |
| 知识提炼 | 从论文中抽取观点、定义与特征，形成可检索、可关联的 Knowledge |
| 配置灵活 | 不同任务可配置不同模型、Agent 与提示词模板，兼顾质量与成本 |
| 本地可控 | 数据尽量保存在本地，便于长期管理、回溯与迁移 |

## 4. 功能需求

### 4.1 文献管理

#### 4.1.1 文献入库

**需求描述：**
用户可通过多种方式将论文纳入平台管理，并在入库阶段自动完成论文处理链路。

**详细需求：**

- 支持单篇手动入库，输入方式至少包括 DOI、URL、标题。
- 支持批量入库能力，适用于后续基于搜索结果、推荐结果或外部列表的一次性导入。
- 入库时支持用户指定论文所属分类，分类采用“方向 / 子方向”层级结构。
- 入库后系统自动执行以下处理流程：
  1. 获取或下载 `paper.pdf`
  2. 执行 PDF 原始解析
  3. 生成解析源文档
  4. 使用 LLM 对解析结果进行清洗、修复与优化
  5. 按章节进行结构切分
  6. 生成 `note`
  7. 完成元数据、文件与状态入库
- 入库时自动记录论文标题、作者、发表年份、来源、DOI、URL、分类、处理状态等信息。
- 论文资源至少包含以下核心产物：
  - `paper.pdf`
  - 解析源文档
  - 优化解析文档
  - `note`

**验收标准：**

- 用户可通过 DOI、URL 或标题完成单篇入库。
- 论文在入库后具备完整处理状态记录。
- 核心论文资产能够在本地工作区中稳定落盘，并可通过系统检索与读取。

---

#### 4.1.2 文献分析

**需求描述：**
论文入库后，系统自动生成研究笔记，并支持用户在同一文档基础上继续修改。

**详细需求：**

- 系统基于章节化后的论文内容自动生成 `note`。
- `note` 至少包含以下信息：
  - 研究问题
  - 核心方法
  - 主要贡献
  - 实验设置与结果摘要
  - 局限性或值得关注的问题
- 用户可直接在 `note` 上继续编辑，无需维护独立的人工笔记文档。
- 系统记录 `note` 的以下状态信息：
  - 自动生成状态
  - 人工编辑状态
  - 最后更新时间
- 已发生人工修改的 `note` 在再次自动生成时，系统需要提供明确的覆盖策略，避免无提示覆盖用户内容。
- 在论文分析阶段，系统可同步抽取数据集、实验设置、关键指标等可结构化信息，为后续 `Dataset`、`Project`、`Knowledge` 能力提供输入。

**验收标准：**

- 每篇入库论文都可生成统一结构的 `note`。
- 用户能够直接编辑 `note`，并保留人工修改记录。
- 重新生成与人工修改之间的边界清晰，状态信息可查询。

---

#### 4.1.3 论文推荐

**需求描述：**
系统基于用户已有研究方向、已入库论文和历史筛选行为推荐相关论文。

**详细需求：**

- 支持基于已入库论文内容与分类推荐候选论文。
- 推荐结果包含基本元数据、推荐理由与入库入口。
- 推荐结果与正式论文库分离管理，避免直接污染正式资源。

**验收标准：**

- 推荐结果可独立查看、筛选与入库。
- 推荐理由与目标论文存在明确关联。

---

#### 4.1.4 Zotero 集成

**需求描述：**
平台预留与 Zotero 的集成能力，用于后续导入已有文献资产。

**详细需求：**

- 支持通过 `Zotero MCP` 读取条目、Collection、元数据与附件信息。
- 首阶段以单向导入为主，后续视能力扩展双向同步。
- 导入后的论文进入平台统一的文献入库与分析链路。

**验收标准：**

- 集成能力可读取外部 Zotero 数据源并执行导入。
- 导入结果可追溯到对应的 Zotero 条目。

---

### 4.2 每日论文推送

**需求描述：**
系统定期收集指定领域的新论文，生成候选清单，供用户筛选后入库。

**详细需求：**

- 系统每日定时自动抓取 `Arxiv` 最新发布的论文
- 对每篇论文自动生成摘要分析与价值评分
- 推送结果以列表形式呈现，用户可快速浏览标题与摘要
- 用户筛选操作：
  - **入库**：将选中论文正式纳入论文库，并触发自动分析文档生成
  - **忽略**：不选择的论文次日自动清除，不占用存储空间
- 支持按评分、研究方向等维度对推送列表进行筛选排序

**验收标准：**

- 候选论文列表具备筛选、排序与入库入口。
- 入库操作可直接接入正式论文处理链路。

---

### 4.3 数据集管理

**需求描述：**
平台管理论文中涉及的数据集信息，并为后续横向比较提供基础。

**详细需求：**

- 支持记录数据集名称、任务类型、规模、来源等信息。
- 支持将论文解析阶段抽取的数据集信息落入数据库。
- 支持查看某个数据集关联的论文与实验结果。

**验收标准：**

- 数据集能够与论文建立稳定关联。
- 数据集视角下可查看对应论文列表及结果摘要。

---

### 4.4 Knowledge 管理（结构化知识提炼）

平台中的 Knowledge 资源用于沉淀论文中的观点、定义、特征及其关系，形成可检索、可关联、可复用的结构化知识集合。

#### 4.4.1 观点提炼

**需求描述：**
观点并非对整篇文章的摘要总结，而是专门提取**论文作者针对当前领域、现有方法或特定问题发表的明确看法、批判或主张**。这些观点应具有强烈的“可引用性”（Citable），方便用户在撰写自己论文的 Introduction 或 Related Work 时直接作为引用素材。

**详细需求：**
- 每条观点必须是作者的一个明确论断（Claim），而非中性的事实说明。需按以下逻辑分类归档：
  - **领域现状评述**：作者对当前研究趋势、现实需求或该领域重要性的主观评价（如“XX 是当前阻碍落地的最大瓶颈”）。
  - **对现有方法的批判**：作者对前人工作指出的具体缺陷或抨击（如“现有方法在 XX 场景下会失效，因为忽略了 XX 因素”）。
  - **核心启发/主张**：作者在方法论层面提出的核心假说或洞见（如“我们认为，引入 XX 机制才能根本上解决 XX 问题”）。
  - **关键性结论**：打破常规认知或具有强主导性的实验发现（如“实验证明，XX 特征在提升鲁棒性上比 XX 更具决定性”）。
- 结构化字段需包含：**观点标题、观点类型、中文简要概括、英文原文依据（必填，确保引用的准确与原汁原味）**。
- **引用对象溯源（极其重要）**：当作者的批判或评述明确指向某篇参考文献时（如“相比于 [12] 提出的方法...”），系统必须准确识别并提取该被引用对象，标注在结构化字段中，形成 `[本文] -> 批判/评价 -> [参考文献]` 的关联网络。
- 用户可对自动生成的观点进行编辑、合并、删除、重分类和补充备注。

#### 4.4.2 定义与特征提取

**需求描述：**
从论文中结构化地提取客观的、具有复用价值的领域名词、数学定义、特征描述及评估标准，辅助用户快速扫盲和搭建代码工程。

**详细需求：**
- 系统需自动识别并提取以下类型的客观定义信息：
  - **核心概念/术语定义**：论文中重点介绍的领域专有名词及其学术解释。
  - **任务定义类**：说明某类研究任务的学术定义、目标、输入规范、输出形式及边界条件。
  - **特征定义类**：说明特定领域内关键特征的物理含义、多模态语义或数学公式表达。
  - **特征提取类**：说明提取上述特征所采用的具体算法、网络结构（如 Transformer Encoder 结构设计）或数据处理流程。
  - **评估指标定义**：论文中使用的性能评估指标的数学计算方式或度量逻辑。
  - **数据构建/标注规则**：针对数据集类论文，提取其数据采集、数据清洗或人工标注的具体规范与准则。
- 结构化字段需包含：**定义标题、定义类型、中文详细解释/公式描述/流程总结**（*注：该类别的重点在于对客观知识的准确归纳与解释，不强制要求保留“英文原文依据”字段*）。
- 用户可对提取的定义进行勘误、补充笔记或归类。

**验收标准：**
- 可准确从论文中抽取上述两类结构化知识。
- 用户可利用观点库梳理研究脉络；利用定义库快速查找“某个任务是什么”、“某种特征怎么提取”、“指标怎么计算”。

- 定义与特征条目能够稳定落库。
- 条目能够支持跨论文复用与对比。

### 4.5 实验规划辅助

**需求描述：**
基于论文、课题与已有知识，为用户提供实验规划参考。

**详细需求：**

- 基于相关论文抽取实验设置、评价指标、对比基线与数据集信息。
- 支持结合当前课题目标生成实验建议。
- 支持以课题为单位记录实验假设、规划与阶段结论。

**验收标准：**

- 用户能够围绕单个课题查看实验规划建议。
- 规划建议可追溯到相关论文与知识来源。

---

### 4.6 课题管理

**需求描述：**  
提供完整的课题生命周期管理能力，支持从立项到结题的全程记录与辅助。

本模块负责平台中的 Project 资源管理，为实验规划、实验结果分析、组会汇报准备与论文写作提供统一上下文。

#### 4.6.1 课题结构

每个课题至少包含以下六个核心模块：

- `overview`
- `related-work`
- `method`
- `experiment`
- `conclusion`
- `manuscript`

各模块职责如下：

**① Overview**

| 项目 | 说明 |
|------|------|
| 目标定位 | 作为课题首页，用于快速呈现研究目标、研究范围、阶段状态与整体进展 |
| 自动化能力 | 自动汇总当前课题已关联的论文数量、关键论文、相关数据集、Knowledge 条目、最近任务与阶段性进展 |
| 人工职责 | 用户补充研究目标、里程碑、阶段计划、重点风险、待验证问题与优先事项 |
| 主要输出 | 课题概览、阶段目标、关键资源清单、近期进展摘要、下一步计划 |

**② Related Work**

| 项目 | 说明 |
|------|------|
| 目标定位 | 维护与课题直接相关的论文材料、方法脉络与对比分析 |
| 自动化能力 | 基于课题关联论文自动整理相关工作候选集合，按主题、方法路线、应用场景或基线类型进行归类 |
| 人工职责 | 用户确认纳入范围，补充遗漏论文，调整分类结构，并加入主观判断与取舍理由 |
| 主要输出 | 相关工作综述、论文分类结构、代表性工作摘要、方法差异分析、引用候选清单 |

**③ Method**

| 项目 | 说明 |
|------|------|
| 目标定位 | 沉淀课题的方法设计、核心假设、技术路线与贡献点 |
| 自动化能力 | 从相关论文中提取现有方法的局限性、潜在改进方向与可复用模块，辅助生成方法草稿 |
| 人工职责 | 用户明确自身方法的核心设计、模块拆解、关键创新点、约束条件与适用边界 |
| 主要输出 | 方法说明、技术路线、模块设计、贡献点草稿、方法风险与假设说明 |

**④ Experiment**

| 项目 | 说明 |
|------|------|
| 目标定位 | 记录实验规划、实验执行、结果分析与问题复盘 |
| 自动化能力 | 基于相关论文与课题目标生成实验方案建议，并在录入结果后辅助完成结果总结与异常分析 |
| 人工职责 | 用户录入实验结果、确认评价指标、记录失败尝试、解释异常现象并决定后续实验方向 |
| 主要输出 | 实验计划、主实验与消融实验记录、结果分析、失败原因、后续实验建议 |

**⑤ Conclusion**

| 项目 | 说明 |
|------|------|
| 目标定位 | 沉淀阶段性研究结论、关键判断与下一阶段决策 |
| 自动化能力 | 基于 `related-work`、`method`、`experiment` 中的内容汇总生成阶段结论草稿 |
| 人工职责 | 用户审核结论、修正表述、保留关键决策，并明确后续行动项 |
| 主要输出 | 阶段性总结、方法有效性判断、经验沉淀、未解决问题、下一步计划 |

**⑥ Manuscript**

| 项目 | 说明 |
|------|------|
| 目标定位 | 作为课题阶段性产出，持续沉淀可迭代、可投稿的论文草稿 |
| 自动化能力 | 基于前述五个模块自动组织全文结构，生成章节草稿，补齐章节衔接与引用占位 |
| 人工职责 | 用户逐章审核、改写、锁定已确认内容，补充图表说明、实验细节与投稿要求相关内容 |
| 主要输出 | 结构化的论文草稿工作区，包含各章节草稿、整篇预览与版本信息 |

#### 4.6.2 课题通用需求

- 支持创建、编辑、归档课题，并维护课题名称、研究方向、阶段状态、更新时间等基础信息。
- 支持在课题下关联任意已入库论文，并标注关系类型，至少包括：
  - `related_work`
  - `baseline`
  - `inspiration`
  - `method_reference`
  - `experiment_reference`
- 支持在课题下关联 Knowledge、Presentation、数据集与实验记录等资源。
- 支持以课题为单位手动触发 LLM 任务，至少包括：
  - 相关工作整理
  - 方法草稿生成
  - 实验规划建议
  - 结果总结
  - 阶段结论汇总
- 支持模块级独立维护，允许用户仅对单一模块进行更新或重生成。
- 支持在模块内部保留来源追溯信息，明确当前内容来自哪些论文、哪些实验记录、哪些 Knowledge 条目。
- 支持在课题范围内维护阶段状态，建议至少包括：
  - `planning`
  - `researching`
  - `experimenting`
  - `writing`
  - `archived`
- `overview` 中的统计与近期进展应在课题关联论文、实验记录或汇报任务更新后自动刷新。
- `manuscript` 与前置模块保持关联，重生成时应支持保留已确认内容或锁定片段。
- 首阶段文档统一采用 Markdown 存储，编辑可先依赖第三方工具完成。

**验收标准：**

- 六个核心模块均可独立维护，且具备明确职责边界。
- 课题与论文、Knowledge、数据集、汇报任务之间的关联关系可追溯。
- 围绕课题触发的自动化任务具备明确状态、输入来源与输出位置。
- 课题下的自动生成内容不会无提示覆盖已确认的人为内容。

#### 4.6.3 PPT 制作（组会汇报）

**需求描述：**
基于课题与论文积累的内容生成组会汇报材料。

**详细需求：**

- 支持在单个课题下创建 `Presentation` 资源。
- 支持为每次汇报选择关联论文、实验结果、阶段结论等输入材料。
- 系统自动生成汇报大纲，并进一步生成逐页草稿。
- 汇报资源至少包含以下产物：
  - 大纲或中间草稿文件
  - 逐页内容草稿
  - 导出的 PPT 文件
- 汇报草稿支持继续编辑与再次导出。

**验收标准：**

- 用户能够围绕单个课题创建并维护多次汇报任务。
- 汇报草稿中的关键内容可追溯到对应论文、实验或课题模块。

### 4.7 论文写作

**需求描述：**
基于课题管理中沉淀的内容生成并维护论文草稿。

**详细需求：**

- 支持针对以下章节分别提供写作辅助：
  - Abstract
  - Introduction
  - Related Work
  - Method
  - Experiment
  - Conclusion
- 支持将各章节内容汇总为完整 `manuscript`。
- 支持用户编辑、重生成、锁定章节与人工修改内容。
- 支持一致性检查，包括术语、贡献点、数据集名称、指标、引用关系等。

**验收标准：**

- 各章节可独立生成与维护。
- 已确认内容在重新生成时具备明确保护机制。
- 章节内容与课题事实信息保持一致。

### 4.8 投稿 & Rebuttal

**需求描述：**
辅助用户完成投稿准备与审稿意见回复。

**详细需求：**

- 支持整理投稿所需材料。
- 支持针对审稿意见生成回复草稿。
- 支持多轮修改与版本管理。

**验收标准：**

- 投稿与回复材料可结构化管理。
- 回复内容与论文内容、实验结果保持一致。

---

### 4.9 录用后处理

**需求描述：**
辅助用户处理论文录用后的后续事项。

**详细需求：**

- 支持 Camera-ready 准备。
- 支持后续宣传材料生成，如海报、演示稿等。
- 支持基于既有论文与课题材料生成后续输出。

**验收标准：**

- 录用后材料可在统一空间中管理与生成。

---

### 4.10 会议追踪

**需求描述：**
集中管理关注会议的关键时间节点与投稿要求。

**详细需求：**

- 支持记录多项会议基本信息与时间节点。
- 支持记录摘要截止、全文截止、通知时间、Camera-ready 时间等。
- 支持定期自动更新会议信息。
- 支持在关键时间点前进行提醒。

**验收标准：**

- 用户可查看已关注会议的关键日程。
- 时间节点提醒与更新状态可查询。

## 5. 知识图谱需求

**需求描述：**
以图谱形式呈现论文、课题、数据集、Presentation 与 Knowledge 之间的关联关系。

### 5.1 图谱类型

| 图谱名称 | 节点类型 | 关系说明 |
|----------|----------|----------|
| 数据集 ↔ 论文 | 数据集、论文 | 论文使用了某个数据集；数据集被哪些论文使用 |
| 论文 ↔ 课题 | 论文、课题 | 论文是某课题的相关工作、方法参考、实验基线或启发来源 |
| Knowledge ↔ 论文 | Knowledge、论文 | Knowledge 来源于某论文，或被某论文支持、定义、反驳 |
| Presentation ↔ 课题 | Presentation、课题 | 某次汇报属于某课题，汇报内容来源于课题模块与论文材料 |

### 5.2 可视化需求

- 支持图形化浏览与搜索定位。
- 支持按节点类型、关系类型筛选。
- 支持从图谱跳转到对应资源详情。
- 图谱随底层数据变化自动更新。

**验收标准：**

- 图谱可用于查看核心资源之间的关系。
- 节点与边具备基本的筛选与跳转能力。

## 7. 非功能需求

| 类别 | 需求 |
|------|------|
| 可用性 | 核心流程应简洁清晰，适合个人研究者快速上手 |
| 本地优先 | 论文文件、解析结果、研究笔记与数据库优先保存在本地 |
| 存储约束 | 首阶段需考虑本地存储有限的现实条件，避免无控制地积累冗余产物 |
| 可扩展性 | 模型、Agent、提示词模板与分析流程支持后续扩展 |
| 稳定性 | 自动化任务失败时需返回明确状态，不得破坏已有数据 |
| 可追溯性 | 研究笔记、知识条目、汇报草稿和课题结论需保留来源信息 |

## 9. 需求优先级

### P0（核心，必须实现）

- 文献入库（批量 + 手动，重点支持 DOI、URL、标题导入）
- 论文处理链路：下载、解析、LLM 优化解析、章节切分、`note` 生成、入库
- `note` 的人工编辑支持，以及编辑状态与最后更新时间记录
- 课题管理（六个核心模块）
- 多模型独立配置
- LLM / Agent 基础配置与可用性管理

### P1（重要，应当实现）

- 数据集管理（前期可先提取信息落库，后续再完善管理视图）
- 每日论文推送与筛选
- Knowledge 管理（观点 / 定义 / 特征）
- 知识图谱（含 Knowledge 多类关系）
- 实验规划辅助
- PPT 制作（组会汇报）
- 论文写作辅助
- 会议追踪
- 提示词模板管理

### P2（增强，有条件实现）

- 论文推荐
- 知识图谱可视化交互增强
- 投稿 & Rebuttal 辅助
- 录用后处理辅助
- Zotero 集成（先单向导入，后续双向同步）

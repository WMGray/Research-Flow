# Research-Flow 用户需求文档

## 1. 目标用户

Research-Flow 面向计算机方向研究者，目标是把已经验证过的 Paper 工作流沉淀为一个真正可运行、可维护、可扩展的前后端项目。

当前优先用户诉求：

- 用本地目录稳定管理论文资料，而不是散落在临时脚本和手工文件夹中。
- 对 Search / Curated / Library 三段式流程建立统一视图。
- 能对单篇论文执行导入、迁移、状态追踪和 note 模板生成。
- 能对 PDF 进行程序式解析，生成可检查的文本、结构和图片产物。
- 能在首页和各流程页面快速看到待处理论文、近期批次和当前库的总体状态。
- 能先人工完成分类、确认和质量判断，不依赖 LLM Provider。

## 2. 本轮范围：Paper 全流程

### 2.1 后端

- 基于 `data/` 建立本地文献库真源。
- 支持 `Discover / Acquire / Library / templates` 四类目录。
- 支持 `scan / ingest / migrate / note-template / parse-pdf` 等脚本能力。
- 使用 JSON 暂存批次、队列、论文状态和解析结果索引。
- 暴露 Home / Discover / Acquire / Library / Paper Detail / Config 所需接口。
- 实现程序式 PDF parser，不接入 LLM 自动判断。

### 2.2 前端

- 建立前后端分离壳。
- 实现 HomePage。
- 补齐 Discover、Acquire、Library、Paper Detail、Config 页面。
- 页面先以可维护的信息架构和真实 API 对接为目标，不追求复杂交互。

### 2.3 数据

- 测试样本可来自现有 `search / curated / Papers / note template`。
- 不直接改动原始研究目录。
- 正式运行时以仓库根 `data/` 为唯一主存储。
- 软件完成后再支持用户在 Config 中指定外部数据根目录。

## 3. 明确不做

- Celery / Redis / worker
- 数据库持久化
- skills 自动调用
- LLM Provider 管理
- 自动化 LLM 分类
- Project / Dataset / Knowledge / Presentation 完整模块
- 多用户权限和远端同步

## 4. 验收重点

- 后端服务能读取真实 `data/` 内容并返回 HomePage 汇总。
- `ingest / migrate` 不会静默覆盖已有论文目录。
- `needs-pdf / needs-review / failed / processed` 状态能进入 dashboard 统计。
- PDF parser 能对真实 `paper.pdf` 生成可复用产物，并记录失败原因。
- Discover / Acquire / Library / Paper Detail / Config 能用真实 API 数据渲染。
- 即使不启动数据库和 LLM 服务，Paper 全流程仍可运行。

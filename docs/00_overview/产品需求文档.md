# Research-Flow 产品需求文档

## 1. 文档说明

### 1.1 编写目的

本文档用于定义 Research-Flow 的产品定位、范围边界、核心资源、关键能力、阶段目标与验收标准，作为后续架构设计、后端实现与路线图制定的依据。

### 1.2 适用范围

本文档适用于当前阶段的产品设计与后端实现，不覆盖详细前端交互设计与视觉设计。

---

## 2. 产品定位

### 2.1 产品定义

Research-Flow 是面向计算机方向 PhD 与科研工作者的本地优先研究管理平台，用于统一管理论文、课题及其衍生研究资产。

### 2.2 产品边界

本产品当前边界如下：

- 面向科研管理，不承担软件开发代码管理、代码托管与工程协作职责。
- 重点覆盖论文入库、解析、研究笔记生成、课题沉淀、模型配置与自动化任务管理。
- 首阶段采用本地优先方案，数据与文件保存在本地文件系统和本地数据库中。
- 首阶段不引入 OSS 等远程对象存储。

### 2.3 目标用户

- 计算机方向博士生
- 科研工作者
- 以个人本地单机使用为主的研究者

### 2.4 核心价值

- 将论文处理链路标准化，降低文献整理成本。
- 将论文输出稳定沉淀到课题空间，减少重复整理。
- 通过可配置的 LLM 与 Agent 能力，提高研究过程中的自动化辅助水平。
- 在本地环境中沉淀长期可复用的研究资产。

### 2.5 非目标

以下能力不属于当前阶段目标：

- 软件项目开发管理
- 多人协作与权限系统
- 云存储与跨设备同步
- 首阶段内置完整 Markdown 编辑器
- Zotero 深度同步

## 3. 核心资源模型

### 3.1 资源优先级

当前资源优先级如下：

`Paper > Project > Presentation > Knowledge > Dataset`

### 3.2 核心资源

**Paper**

- 产品当前阶段的主入口资源。
- 承担论文导入、下载、解析、优化、分节、研究笔记生成与元数据管理。

**Project**

- 承担课题级材料沉淀与后续研究活动组织。
- 在 `Paper` 主链路稳定后接入，使用论文解析结果、分节结果与 LLM 分析结果作为输入。

### 3.3 衍生资源

**Presentation**

- 用于阶段性汇报输出。
- 当前阶段仅保留产品定义，不进入正式交付范围。

**Knowledge**

- 当前阶段先作为结构化数据管理能力存在。
- 首阶段重点是从论文中抽取并落库，不要求完整独立资源工作台。

**Dataset**

- 当前阶段先承接论文解析与实验抽取结果。
- 后续再补独立管理视图与横向比较能力。

### 3.4 资源关系

- `Paper` 为 `Project` 提供主要输入材料。
- `Project` 消费 `Paper` 的章节结果、结构化抽取结果与 `note`。
- `Knowledge` 主要来源于 `Paper` 解析与分析过程。
- `Presentation` 可独立存在，也可关联多个 `Project`，或仅引用多篇 `Paper` 与实验记录。

---

## 4. 当前阶段产品范围

### 4.1 当前阶段目标

当前阶段目标为在两周内完成单机可用的平台首版，重点建立以下闭环：

1. 单篇论文导入与入库
2. 论文下载、解析、LLM 优化、分节与 `note` 生成
3. 基于已处理论文创建课题并初始化核心模块
4. 结构化 Knowledge 与数据集结果落库
5. 每日论文推送、论文推荐与会议追踪
6. 知识图谱呈现
7. Presentation 中间草稿与 `pptx` 生成
8. 论文写作辅助、投稿与 Rebuttal 辅助、录用后处理辅助
9. 按环节配置模型、Agent 与兜底策略

### 4.2 当前阶段不做

- 批量搜索导入
- Zotero 集成

## 5. Paper 产品需求

### 5.1 输入方式

当前阶段支持以下单篇导入方式：

- DOI
- URL
- 标题

当前阶段不做批量搜索导入。

### 5.2 分类策略

- 论文分类支持在导入时填写。
- 分类可暂时缺省。
- 若用户未补充分类，系统需记录为待补充状态，并生成明确提醒，提示用户后续完善分类信息。

### 5.3 论文处理主链路

当前阶段的标准处理链路如下：

1. 下载或关联 `paper.pdf`
2. 执行 PDF 原始解析
3. 生成解析源文档
4. 执行 LLM 优化解析
5. 进入人工介入修改节点
6. 进行章节切分
7. 执行 LLM 分析
8. 生成 `note`
9. 完成入库

### 5.4 人工介入节点

- 人工介入节点位于“LLM 优化解析”之后、“章节切分”之前。
- 该节点需要支持开关控制。
- 开启时，用户可在继续分节前查看或修改优化后的解析文档。
- 关闭时，系统自动继续执行后续步骤。

### 5.5 核心输出物

每篇论文至少生成以下产物：

- `paper.pdf`
- 解析源文档
- 优化后的解析文档
- 分节结果
- `note`
- 结构化元数据

### 5.6 `note` 策略

- `note` 由系统自动生成。
- 用户可直接在 `note` 上继续修改。
- 系统需要记录 `note` 的生成状态、人工编辑状态与最后更新时间。
- 后续重新生成 `note` 时需遵循明确的保护策略，避免无提示覆盖用户内容。

### 5.7 成功标准

- 用户可通过 DOI、URL 或标题完成单篇导入。
- 单篇论文可稳定走完整条主链路。
- 链路中各步骤状态可查询。
- 未填写分类时系统会给出待补充提醒。

## 6. Project 产品需求

### 6.1 接入时机

`Project` 在 `Paper` 总流程稳定完成后进入当前阶段范围。

原因如下：

- 课题模块依赖论文的章节结果。
- 课题模块依赖论文中的结构化提取结果。
- 课题模块依赖论文的 LLM 分析结果与 `note`。

### 6.2 课题目标

`Project` 用于沉淀和组织与某个研究问题相关的长期材料，为实验规划、阶段总结与论文写作提供统一上下文。

### 6.3 核心模块

每个 `Project` 创建后自动初始化以下六个模块：

- `overview`
- `related-work`
- `method`
- `experiment`
- `conclusion`
- `manuscript`

### 6.4 使用边界

- 当前阶段允许模块自动初始化。
- 当前阶段允许用户手动触发模块级 LLM 任务。
- 当前阶段不要求所有模块都自动生成完整内容，但要具备统一骨架和后续填充能力。

### 6.5 论文关联

`Project` 需要支持与已入库论文建立显式关联，关系类型首阶段固定为少量枚举：

- `related_work`
- `baseline`
- `inspiration`
- `method_reference`
- `experiment_reference`

### 6.6 状态管理

当前阶段采用以下课题状态：

- `planning`
- `researching`
- `experimenting`
- `writing`
- `archived`

### 6.7 成功标准

- 在论文主链路完成后，可基于论文创建课题。
- 课题创建后自动拥有六个核心模块。
- 课题可关联论文并标注关系类型。
- 课题范围内的 LLM 任务可手动触发并返回结果。

## 7. Knowledge 产品需求

### 7.1 当前阶段定位

`Knowledge` 当前阶段只做数据管理，不做完整独立产品页。

### 7.2 当前阶段能力

- 从 `Paper` 分析过程中抽取观点、定义、特征等结构化结果。
- 将抽取结果落入数据库。
- 保留与来源论文的关联关系。
- 为后续 `Project`、`Dataset`、知识图谱与写作能力提供数据基础。

### 7.3 当前阶段不做

- 独立的 Knowledge 工作台
- 复杂的人工编辑界面
- 深度图谱交互

---

## 8. Presentation 产品定义

### 8.1 阶段定位

`Presentation` 当前进入平台首版范围，但优先级低于 `Paper` 与 `Project`。

### 8.2 资源形态

`Presentation` 为独立资源，不要求强绑定单一 `Project`。

允许以下情况：

- 关联多个 `Project`
- 不关联任何 `Project`
- 引用多篇 `Paper`
- 读取用户提供的实验记录 Markdown

### 8.3 目标能力

- 基于输入材料生成中间草稿
- 基于中间草稿生成 `pptx`
- 使用模板驱动生成
- 预留后续命令调用方式

### 8.4 待后续补充

- 模板由后续提供
- 导出命令由后续提供

## 9. LLM / Agent 配置需求

### 9.1 首批接入范围

首批模型来源至少包括：

- 硅基流动
- DeepSeek
- 其他兼容接入的模型来源

### 9.2 配置粒度

配置粒度按处理环节划分，而非仅按模块划分。

当前至少需要支持以下环节独立配置：

- 论文优化解析
- 论文分节后分析
- `note` 生成
- 课题相关工作整理
- 方法草稿生成
- 实验规划建议
- 结果总结
- 结论汇总
- 模型连通性探测

### 9.3 配置能力

每个环节需要支持：

- 主模型选择
- 备用模型选择
- 可用性检测
- 状态查询

### 9.4 兜底策略

- 每个关键环节应支持备用模型兜底。
- 当主模型不可用时，系统可切换到备用模型继续执行。

### 9.5 Agent 策略

当前阶段先做 `Agent` 与任务绑定，不做复杂编排中心。

### 9.6 可观测性

系统需要支持以下模型运行信息：

- 联通状态
- 首字时延
- 最近探测时间
- 错误信息

## 10. P0 / P1 / P2

当前文档中的 `P0 / P1 / P2` 表示当前两周周期内的实现顺序、投入优先级与完成度目标，不表示是否进入当前版本范围。除 Zotero 外，其余能力均进入当前周期。

### 10.1 P0

- 单篇论文导入
- 论文下载、解析、LLM 优化、人工介入节点、分节、LLM 分析、`note` 生成与入库
- 论文分类待补充提醒
- `Project` 创建与六模块初始化
- 论文与课题关联
- 按环节配置模型与备用模型
- LLM / Agent 基础绑定与状态管理

### 10.2 P1

- Knowledge 进一步管理能力
- `Dataset` 独立管理视图
- 每日论文推送
- Presentation 中间草稿与 `pptx` 生成
- 知识图谱呈现
- 提示词模板管理
- 会议追踪
- 论文写作辅助

### 10.3 P2

- 论文推荐
- 知识图谱增强交互
- 投稿与 Rebuttal 辅助
- 录用后处理辅助

### 10.4 当前周期明确延后

- Zotero 集成

## 11. 成功标准

当前阶段的产品成功标准如下：

- 单篇论文可从 DOI、URL 或标题入口完成稳定处理闭环。
- 论文处理链路覆盖下载、解析、LLM 优化、人工介入、分节、LLM 分析与 `note` 生成。
- `Project` 可在论文处理结果基础上创建并投入使用。
- `Knowledge` 抽取结果能够稳定落库并保留来源关系。
- `Dataset` 结果能够稳定落库并支持基础管理。
- 模型配置能够按环节选择主模型和备用模型，并可查看连通状态与首字时延。
- 每日论文推送、论文推荐、会议追踪、Presentation 输出、论文写作辅助、投稿与 Rebuttal 辅助、录用后处理辅助均形成首版可操作能力。

---

## 12. 约束条件

- 个人项目
- 当前计划周期为两周
- 本地存储有限
- 当前阶段不考虑 OSS
- 当前阶段不做批量搜索导入

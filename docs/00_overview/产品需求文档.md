# Research-Flow 产品需求文档

## 1. 产品定位

Research-Flow 是一个面向研究流程的本地优先工作台。第一阶段只聚焦 Paper 全流程，把“论文发现、筛选、获取、解析、入库、整理、查看、配置”这条链路做成可长期维护的软件能力。

本阶段不是完整科研平台，也不是自动化 Agent 系统。它优先解决 Paper 资产从零散目录、临时脚本和人工整理流程，收敛到统一 `data/` 真源的问题。

## 2. 产品目标

- 统一接住现有 Paper 处理资产，并迁移到仓库根 `data/`。
- 提供清晰的 Discover -> Acquire -> Library -> Paper Detail 状态流转。
- 支持程序式 PDF parser，生成可检查、可重跑、可追踪的解析产物。
- 使用 JSON / Markdown / 文件系统暂存状态，暂不引入数据库。
- 让前端具备 Home / Discover / Acquire / Library / Paper Detail / Config 的可维护页面边界。
- 为后续自定义数据目录、数据库、LLM Provider 和 Project 模块保留扩展点。

## 3. 核心流程

```text
Discover
  -> search batch
  -> candidate review
  -> keep / reject

Acquire
  -> curated queue
  -> PDF availability check
  -> metadata normalization
  -> manual confirmation when needed

Library
  -> categorized paper directory
  -> metadata.json
  -> paper.pdf
  -> note.md
  -> parser outputs

Paper Detail
  -> metadata review
  -> PDF parse status
  -> note / refined / images
  -> manual fixes

Config
  -> data root display
  -> parser options
  -> path checks
```

## 4. 阶段性取舍

本阶段明确不做：

- 数据库持久化
- Celery / Redis / worker
- LLM Provider 管理
- Agent / Skill 自动调用
- Project / Dataset / Knowledge 完整模块
- 多用户、权限、远端同步

本阶段必须做：

- 根 `data/` 作为唯一真源
- JSON 暂存状态和索引
- Paper 全流程页面与接口边界
- 程序式 PDF parser 与解析产物管理
- 人工确认节点
- 可重复运行的 CLI / API / 测试

## 5. 成功标准

- 可以用脚本把外部论文目录纳入 `data/Library/`。
- 可以从 Discover 批次进入 Acquire 队列，再进入 Library。
- 可以检测缺失 PDF、冲突目录、解析失败和人工待审状态。
- 可以对单篇论文执行程序式 PDF 解析，并保存结构化产物。
- 可以基于模板生成 note，并在 Paper Detail 中查看关键产物。
- 可以通过 HomePage 作为统一入口查看全局状态。
- 可以通过 Config 页面查看当前数据根、路径健康和 parser 配置。
- 所有核心状态都能通过文件系统和 JSON 恢复，不依赖运行时内存。

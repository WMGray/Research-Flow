# Research-Flow 路线图

## 1. 文档说明

### 1.1 编写目的

本文档用于定义 Research-Flow 的平台建设节奏、阶段目标、里程碑拆分与延后项，作为当前阶段产品推进和实施排期的依据。

### 1.2 文档定位

本路线图从平台视角组织，不按后端开发任务或技术子系统拆分。路线图关注的是平台在各阶段向用户提供的能力与可演示成果，而非具体实现细节。

---

## 2. 规划原则

### 2.1 平台优先

Research-Flow 当前按研究管理平台进行建设，路线图围绕资源能力、用户任务与平台可感知成果制定，不以单一后端模块为主轴。

### 2.2 当前周期目标

当前周期为个人项目的首轮集中交付阶段，目标是在较短时间内先完成整个平台的一版可用能力，优先保证流程打通，不以代码质量和长期工程形态作为首要目标。

### 2.3 优先级含义

当前优先级决定的是实现顺序与投入重心，不代表低优先级能力不进入本轮路线图。除 Zotero 外，其余核心能力均进入当前平台首版范围。

### 2.4 明确延后项

当前阶段唯一明确延后的能力为 Zotero 集成。

---

## 3. 当前周期总目标

### 3.1 平台交付目标

本轮目标是在当前周期内完成一版可演示、可操作、前后端流程完整的平台版本，覆盖以下核心能力：

- 论文导入、下载、解析、LLM 优化、分节、分析与 `note` 生成
- 课题创建、六模块初始化、论文关联与模块级手动分析触发
- Knowledge 与 `Dataset` 数据落库及基础管理
- 模型状态查看、按环节切换配置、主模型与备用模型兜底
- 每日论文推送
- 知识图谱呈现
- Presentation 中间草稿与 `pptx` 生成能力
- 论文写作辅助、投稿与 Rebuttal 辅助、录用后处理辅助

### 3.2 当前周期平台边界

当前周期的平台首版以“先跑通全链路”为目标，具备以下特征：

- 支持单篇论文导入，不做批量搜索导入
- 支持本地优先运行，不考虑 OSS
- 支持人工在关键节点介入，不追求全自动闭环
- 支持较粗粒度但完整的平台功能覆盖

---

## 4. 关键演示场景

当前路线图围绕以下四类关键演示场景组织：

### 4.1 论文主链路演示

- 用户输入 DOI、URL 或标题导入论文
- 系统完成下载、解析、LLM 优化、人工介入、分节、LLM 分析
- 系统生成并展示 `note`

### 4.2 课题分析演示

- 用户创建课题
- 用户关联一批论文
- 用户手动触发相关工作整理、论文对比与实验规划分析

### 4.3 模型配置演示

- 用户查看各模型联通状态与首字时延
- 用户按不同环节切换模型配置
- 系统在主模型失败时切换备用模型

### 4.4 平台视图演示

- 系统展示 Knowledge 落库结果
- 系统展示知识图谱
- 系统生成 Presentation 中间草稿与 `pptx`

## 5. 里程碑规划

### M0 需求与平台骨架冻结

**目标**

- 完成平台顶层定义、资源主次关系与当前周期范围冻结

**平台范围**

- 完成 URD
- 完成 PRD
- 完成路线图
- 明确当前周期平台目标、延后项与关键演示场景

**用户可感知成果**

- 平台目标和阶段目标明确
- 后续实现不再围绕“是否做某能力”反复摇摆

**完成标准**

- 核心资源、主链路、优先级与当前周期范围在文档中形成统一口径

---

### M1 Paper 主链路打通

**目标**

- 建立平台的主入口能力，让用户完成单篇论文入库闭环

**平台范围**

- 单篇论文导入
- PDF 下载或挂接
- 原始解析
- LLM 优化解析
- 人工介入开关
- 分节
- LLM 分析
- `note` 生成
- 入库状态记录
- 分类待补充提醒

**用户可感知成果**

- 用户可从 DOI、URL 或标题开始导入论文
- 用户可看到完整处理过程和结果文件
- 用户可得到可继续修改的 `note`

**依赖关系**

- 这是 `Project`、`Knowledge`、`Presentation` 的前置基础

**完成标准**

- 单篇论文链路稳定跑通
- 链路状态可查询
- `note` 可生成并可编辑

---

### M2 Project 能力接入

**目标**

- 在 `Paper` 链路完成后，将论文结果沉淀到课题空间中

**平台范围**

- 创建课题
- 自动初始化六模块骨架
- 关联论文
- 标注论文关系类型
- 手动触发相关工作整理
- 手动触发实验规划建议
- 手动触发阶段总结类任务

**用户可感知成果**

- 用户可围绕一个课题组织多篇论文
- 用户可在课题下触发 related work 对比与实验规划分析
- 论文处理结果不再停留在单篇论文层面，而能进入长期研究空间

**依赖关系**

- 依赖 `Paper` 的分节结果、结构化结果与 `note`

**完成标准**

- 课题可创建并可关联论文
- 六模块骨架可初始化
- 至少支持相关工作整理与实验规划两个关键触发场景

---

### M3 配置中心、Knowledge 与 Dataset 数据化

**目标**

- 建立平台级配置能力和结构化知识数据基础

**平台范围**

- 查看模型联通状态
- 查看首字时延
- 按环节切换模型配置
- 配置主模型与备用模型
- Agent 与任务绑定
- Knowledge 抽取结果落库
- 数据集抽取结果落库
- `Dataset` 基础管理

**用户可感知成果**

- 用户可知道哪些模型可用、哪个环节正在用什么模型
- 用户可切换配置并验证状态
- 平台开始从“文档结果”走向“结构化结果”

**依赖关系**

- 依赖 `Paper` 与 `Project` 的核心流程基本稳定

**完成标准**

- 模型状态页可用
- 环节级配置可切换
- 备用模型兜底可配置
- Knowledge 和数据集结果可稳定落库
- `Dataset` 具备基础可查与可关联能力

---

### M4 Platform 展示能力补齐

**目标**

- 补齐平台首版中的可视化与汇报能力，使平台从“能处理”进入“能展示与输出”

**平台范围**

- 知识图谱呈现
- Presentation 中间草稿生成
- `pptx` 导出
- 提示词模板管理

**用户可感知成果**

- 用户可以查看知识图谱
- 用户可以围绕多篇论文、多个课题或实验记录生成汇报草稿
- 用户可以导出 `pptx`
- 用户可以管理提示词模板

**依赖关系**

- 依赖前序里程碑提供的论文、课题、Knowledge 与配置能力

**完成标准**

- 知识图谱可呈现主要关系
- Presentation 可产出中间草稿和 `pptx`
- 提示词模板形成基础能力

---

### M5 平台增强能力补齐

**目标**

- 补齐当前周期剩余的增强能力，使平台首版覆盖完整的研究管理范围

**平台范围**

- 每日论文推送与筛选
- 论文推荐（P2）
- 会议追踪（P2）
- 论文写作辅助
- 投稿与 Rebuttal 辅助
- 录用后处理辅助
- 知识图谱基础交互增强

**用户可感知成果**

- 用户可以查看每日候选论文并执行筛选或入库
- 用户可以获得论文写作、投稿答复和录用后材料整理的初稿支持
- 用户可以在知识图谱中执行基础交互查看

**依赖关系**

- 依赖前序里程碑提供的论文、课题、Knowledge、Presentation 与配置能力

**完成标准**

- 每日论文推送链路可以运行
- 论文推荐与会议追踪进入 P2 收尾补齐清单
- 写作、投稿与录用后处理形成首版可操作能力
- 知识图谱具备基础交互能力

---

### M6 当前周期收口与全链路联调

**目标**

- 将各里程碑能力连成完整平台流程，形成当前周期的可演示首版

**平台范围**

- 前后端流程联调
- 关键演示场景串联
- 主要状态与错误路径补齐
- 演示数据与平台说明整理

**用户可感知成果**

- 平台具备从论文导入到课题分析、配置切换、知识图谱展示、汇报输出的整体演示能力

**完成标准**

- 四类关键演示场景均可完成
- 除 Zotero 外的平台首版能力闭环可演示

---

## 6. 当前周期完成定义

若满足以下条件，则可视为当前周期路线图完成：

- 平台首版已覆盖 Paper、Project、Knowledge、`Dataset`、配置中心、知识图谱与 Presentation 的目标能力
- 用户可完成论文主链路演示
- 用户可完成课题关联论文并触发相关工作和实验规划分析
- 用户可查看模型状态并切换配置
- 用户可查看知识图谱并生成 Presentation 结果
- 用户可使用每日论文推送和论文写作辅助的首版能力；论文推荐、会议追踪、投稿与 Rebuttal 辅助、录用后处理辅助进入 P2 收尾补齐

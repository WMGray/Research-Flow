# 方案概述

当前流程中，Discover 负责接受候选文献，Library 负责管理正式文献。但“进入 Library”不等于“已就绪”。一篇文献可能已经入库，却仍然缺少 PDF、尚未解析、refined 产物未审核、LLM note 未生成，或者 note 还没有经过人工确认。

因此需要把 Library 中的文献继续拆成两层含义：

1. **已入库**：文献已经被接受并进入主库，可以被追踪、分类和处理。
2. **已就绪**：PDF、解析产物、refined 审核、LLM note 和 note 审核都已完成，可以作为可靠知识资产使用。

核心原则是：**Library 仍然是唯一主库，但已入库文献必须通过状态和日志明确展示它卡在哪一步，不能把“已接受”和“已就绪”混为一谈。**

# 设计目标

- 不增加大量 tabs，避免把文献管理拆成多套入口。
- 每篇文献只展示一个用户可理解的 `workflow_status`。
- 提供一个轻量的“未完成”视图，集中处理仍需要动作的已入库文献。
- 每篇文献都有可阅读的操作日志，能解释发生了什么、为什么停住、下一步该做什么。
- 将 refined 审核和 LLM note 审核作为硬闸门，未人工通过时不能进入下一关。

# 已入库文献处理流程

完整流程如下：

```text
入库
  -> PDF 准备
  -> PDF parse
  -> refined 生成
  -> 人工审核 refined
  -> LLM note 生成
  -> 人工审核 note
  -> 分类归档
  -> 已就绪
```

| 阶段 | 入口条件 | 关键产物 | 状态变化 | 用户动作 | 失败处理 |
|---|---|---|---|---|---|
| 入库 | Discover 接受或手动导入 | paper 目录、metadata、state | `stage=library` 或 `review_status=accepted` | 查看详情、补充元数据 | 目录冲突或元数据异常进入失败态 |
| PDF 准备 | 文献已入库 | `paper.pdf` | `asset_status=pdf_ready` | 绑定、下载或替换 PDF | 缺失时显示 `待下载 PDF` |
| PDF parse | PDF 可用 | parsed text、sections、images、`refined.md` | `parser_status=parsed` | 执行或重试解析 | 失败时显示 `失败需处理` |
| refined 审核 | parse 成功并生成 refined | `refined.md` | `refined_review_status=pending` | 批准或驳回 refined | 驳回后显示 `Refine 需修改` |
| LLM note 生成 | refined 审核通过 | `note.md` | `note_status=review_pending` 或兼容 pending 状态 | 生成 note | 生成失败显示 `失败需处理` |
| note 审核 | note 已生成 | 审核后的 `note.md` | `note_review_status=pending` | 批准或驳回 note | 驳回后显示 `Note 需修改` |
| 分类归档 | note 审核通过 | Domain / Area / Topic | `classification_status=classified` | 调整分类、移动目录 | 分类为空时保留待分类提示 |
| 已就绪 | 所有关键闸门通过 | 可读文献资产 | `workflow_status=ready` | 阅读、引用、项目复用 | 后续修改会重新进入对应待审状态 |

## 硬闸门规则

1. `parser_status == parsed` 后不能直接进入 note 生成，而是先进入 `refined_review_status == pending`。
2. 只有 `refined_review_status == approved` 才允许生成 LLM note。
3. LLM note 生成后不能直接算作 ready，而是进入 `note_review_status == pending`，或兼容表示为 `note_status == review_pending`。
4. 只有 note 人工审核通过后，文献才允许显示为 `已就绪`。
5. `refined_review_status == rejected` 和 `note_review_status == rejected` 都必须阻断后续自动推进，并提示用户需要修改后重新审核。

# 状态模型

前端不直接把 `asset_status`、`parser_status`、`note_status`、`review_status`、`refined_review_status`、`note_review_status` 全量暴露给用户，而是折叠成一个综合状态 `workflow_status`。

## 用户可见状态

| 综合状态 | 含义 |
|---|---|
| 待下载 PDF | 文献已入库，但还没有可用 PDF |
| 下载中 | 正在获取 PDF |
| 待解析 | PDF 已有，但还没有进入解析流程 |
| 解析中 | 正在执行 PDF 解析 |
| 待审核 Refine | PDF parse 已完成，refined 产物等待人工确认 |
| Refine 需修改 | refined 审核被驳回，需要修正或重新 parse |
| 待生成 LLM Note | refined 已审核通过，下一步可以生成 LLM note |
| 待审核 Note | LLM note 已生成，等待人工确认 |
| Note 需修改 | note 审核被驳回，需要修正或重新生成 |
| 待分类 | note 已审核通过，但 Domain / Area / Topic 尚未完成 |
| 已就绪 | 关键材料和人工审核都已完成，可正常使用 |
| 失败需处理 | 某一步出错，需要人工干预或重试 |

## 内部状态

底层字段继续保留给系统执行和判定使用：

- `stage`
- `asset_status`
- `parser_status`
- `note_status`
- `review_status`
- `refined_review_status`
- `note_review_status`
- `classification_status`

建议约束：

- `refined_review_status`: `pending | approved | rejected`
- `note_review_status`: `pending | approved | rejected`
- `note_status`: 当前兼容 `missing | template | ready`；后续可扩展为 `missing | generated | review_pending | approved | rejected`
- `workflow_status`: 派生展示字段，不作为唯一执行真源

# 推荐折叠规则

可按下面优先级生成 `workflow_status`：

1. 若任一步骤存在不可恢复错误，显示 `失败需处理`。
2. 若缺少 PDF，显示 `待下载 PDF`。
3. 若正在下载 PDF，显示 `下载中`。
4. 若 PDF 已有但未解析，显示 `待解析`。
5. 若正在解析，显示 `解析中`。
6. 若 `parser_status == parsed` 且 `refined_review_status == pending`，显示 `待审核 Refine`。
7. 若 `refined_review_status == rejected`，显示 `Refine 需修改`。
8. 若 `refined_review_status == approved` 且 note 尚未生成，显示 `待生成 LLM Note`。
9. 若 note 已生成但 `note_review_status == pending`，显示 `待审核 Note`。
10. 若 `note_review_status == rejected`，显示 `Note 需修改`。
11. 若 note 审核通过但分类未完成，显示 `待分类`。
12. 若 PDF、parse、refined 审核、note 生成、note 审核和分类都已完成，显示 `已就绪`。

这个规则只让用户看到一个清晰状态，同时保留底层状态树给系统做能力判断。

# 能力判断

页面按钮和批量动作必须遵守状态闸门：

| 能力 | 允许条件 |
|---|---|
| 解析 PDF | `asset_status=pdf_ready` 且 `parser_status != running` |
| 审核 refined | `parser_status=parsed` 且存在 `refined.md` |
| 生成 LLM note | `parser_status=parsed` 且 `refined_review_status=approved` |
| 审核 note | note 已生成，且 note 审核未通过 |
| 标记已就绪 | refined 审核通过、note 审核通过，且关键文件存在 |

任何自动流程都不能绕过人工审核 refined 和人工审核 note 两个关口。

# 日志设计

每篇文献都应维护一条可阅读的事件时间线，放在 Paper Detail 或右侧详情面板中。日志用于解释流程，不替代底层状态字段。

## 存储建议

- 单篇文献目录下新增 `events.jsonl` 或 `logs/events.jsonl`。
- `parser_runs.json` 继续保留 parser 运行详情，不和通用事件日志混合。
- 前端默认展示人类可读摘要，技术详情折叠展示。

## 日志字段

| 字段 | 说明 |
|---|---|
| `timestamp` | 事件发生时间 |
| `event` | 稳定事件名 |
| `actor` | `system`、`user`、`llm` 或具体操作者 |
| `result` | `success`、`failed`、`pending`、`rejected`、`skipped` |
| `message` | 人类可读描述 |
| `technical_detail` | 可折叠技术细节 |
| `next_action` | 系统建议的下一步动作 |

## 核心事件

| 事件 | 触发时机 |
|---|---|
| `accepted_to_library` | 文献进入 Library |
| `pdf_ready` | PDF 已绑定、下载或确认存在 |
| `parse_started` | 开始 PDF parse |
| `parse_succeeded` | parse 成功并写出产物 |
| `parse_failed` | parse 失败 |
| `refined_generated` | refined 产物生成 |
| `refined_review_approved` | 人工批准 refined |
| `refined_review_rejected` | 人工驳回 refined |
| `llm_note_started` | 开始生成 LLM note |
| `llm_note_generated` | LLM note 生成完成，进入待审核 |
| `note_review_approved` | 人工批准 note |
| `note_review_rejected` | 人工驳回 note |
| `classification_updated` | Domain / Area / Topic 更新 |
| `workflow_ready` | 所有关键步骤完成，进入已就绪 |

## 日志示例

```jsonl
{"timestamp":"2026-05-17T10:12:00+08:00","event":"accepted_to_library","actor":"user","result":"success","message":"已从 Discover 接受并进入 Library。","technical_detail":"","next_action":"检查 PDF 是否可用。"}
{"timestamp":"2026-05-17T10:15:00+08:00","event":"parse_succeeded","actor":"system","result":"success","message":"PDF 解析完成，并生成 refined 文档。","technical_detail":"parser=auto; sections=12; figures=6","next_action":"请人工审核 refined 文档。"}
{"timestamp":"2026-05-17T10:20:00+08:00","event":"refined_review_approved","actor":"user","result":"success","message":"refined 文档已人工确认，可用于生成 LLM note。","technical_detail":"","next_action":"生成 LLM note。"}
{"timestamp":"2026-05-17T10:27:00+08:00","event":"llm_note_generated","actor":"llm","result":"pending","message":"LLM note 已生成，等待人工审核。","technical_detail":"template=default","next_action":"请审核 note 内容。"}
```

日志的重点不是堆叠机器字段，而是让用户一眼看懂“刚才发生了什么、为什么停住了、下一步该做什么”。

# 视图设计

不新增复杂的多级页面，只保留轻量入口：

- **Library 默认页**：展示全部文献。
- **未完成视图**：展示 `workflow_status != 已就绪` 的文献。
- **需要人工处理视图**：优先展示 `待审核 Refine`、`Refine 需修改`、`待审核 Note`、`Note 需修改`、`失败需处理`。
- **数量提示**：显示未完成数量和人工待处理数量，帮助用户快速定位。

这个视图的作用不是再分一套分类体系，而是把“需要动作的文献”集中出来。

# 页面交互

## Discover 顶部信息

Discover 顶部暂不使用“已收录”“待处理队列”“分数”“已选中”这类弱指标。后续可换成更有操作价值的流程指标：

- `待审核 Refine`
- `待审核 Note`
- `失败需处理`
- `今日新增`

这些指标直接对应下一步动作，避免展示简单但不可操作的数字。

## Library 列表

每一行建议展示：

- 标题
- 单个 `workflow_status` badge
- 最近一条日志摘要
- 下一步建议
- “查看日志”入口

## Paper Detail

详情页建议包含：

- 状态总览
- 关键文件路径
- refined 审核按钮：批准 / 驳回
- note 审核按钮：批准 / 驳回
- 生成 LLM note 按钮，只有 refined 审核通过后可用
- 日志时间线

## 未完成视图

未完成视图过滤出：

```text
workflow_status != 已就绪
```

排序优先级：

1. `失败需处理`
2. `Refine 需修改`
3. `Note 需修改`
4. `待审核 Refine`
5. `待审核 Note`
6. 其他等待系统动作的状态

# 后续代码落地方向

## 后端

- parser 成功后写入 `refined_review_status=pending`，并追加 `refined_generated` 日志。
- `generate_note` 能力判断改为要求 `parser_status=parsed && refined_review_status=approved`。
- LLM note 生成后进入 note 待审核状态，不直接视为 ready。
- 新增 refined / note 审核 API，并在审核动作中追加事件日志。
- `workflow_status` 可以由后端派生返回，也可以先由前端用统一函数折叠，但规则必须只有一份真源。

## 前端

- Library 和 Paper Detail 显示综合 `workflow_status`。
- Paper Detail 增加日志 timeline。
- Paper Detail 增加 `批准 Refine / 驳回 Refine / 批准 Note / 驳回 Note` 操作。
- 未完成视图按 `workflow_status != 已就绪` 过滤，并优先展示需要人工动作的文献。

## API 建议

- `GET /api/papers/{paper_id}/events`
- `POST /api/papers/{paper_id}/review-refined`
- `POST /api/papers/{paper_id}/review-note`

审核请求至少包含：

- `decision`: `approved | rejected`
- `comment`: 可选人工说明

# 测试方案

## 文档检查

- 已入库文献流程覆盖从入库到已就绪的完整路径。
- refined 审核和 note 审核都被写成硬闸门。
- rejected 状态不会自动进入下一阶段。

## 后端测试

- parser 成功后不直接允许生成 note，必须先 approve refined。
- refined 被 rejected 后，`generate_note` 不可用。
- LLM note 生成后不直接进入已就绪，必须先 approve note。
- note 被 rejected 后，`workflow_status` 显示 `Note 需修改`。
- 日志事件按动作顺序追加，最近事件摘要可读取。

## 前端测试

- `workflow_status` 折叠规则覆盖 PDF 缺失、parse 失败、refined 待审、note 待审、ready 等状态。
- Library 未完成视图能筛出所有未通过最终审核的已入库文献。
- Paper Detail 的审核按钮根据状态正确启用或禁用。

# 落地顺序

1. 先更新文档与接口约定，统一流程语言。
2. 后端补齐状态字段、日志写入 helper 和审核 API。
3. 后端更新 capability gating，确保 note 生成和 ready 状态不能绕过人工审核。
4. 前端接入 `workflow_status`、日志 timeline 和审核按钮。
5. 增加后端与前端逻辑测试。
6. 最后优化 Discover 顶部指标和 Library 未完成视图的排序。

# 结论

这个方案保持 Library 作为唯一主库的定位，不额外制造复杂 tabs，也不会把“已接受”“已入库”和“已就绪”混为一谈。用户看到的是一个明确的状态，一个可收敛的未完成视图，以及一条能讲清楚过程的日志。

最关键的规则是：**refined 必须人工审核通过后才能生成 LLM note；LLM note 必须人工审核通过后，文献才能进入已就绪。**

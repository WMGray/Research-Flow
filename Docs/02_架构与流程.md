# Research-Flow 架构与流程

## 1. 架构目标

1. 本地优先。
2. 前后端分离。
3. 文件和数据库协同。
4. 后台任务可恢复。
5. 导出适配 Obsidian。

## 2. 推荐架构

```text
Frontend
  Dashboard / Table / Detail / Review
Backend
  FastAPI / Service / Task Controller
Storage
  SQLite / File Store / Logs
Worker
  Parse / Classify / Export
```

## 3. 核心实体

1. Paper。
2. Candidate。
3. SearchBatch。
4. Project。
5. Task。
6. ReviewDecision。

## 4. 状态流转

```text
discovered -> curated -> acquired -> needs-review -> library -> understood
```

异常分支：

```text
curated -> needs-pdf
acquired -> failed
needs-review -> archived
```

## 5. 页面结构

1. Home。
2. Papers Overview。
3. Discover。
4. Acquire。
5. Library。
6. Project Detail。
7. Runtime。
8. Settings。

## 6. 任务流程

1. Discover 检索生成 Search Batch。
2. Gate 1 处理 Candidate。
3. Curated 队列绑定 PDF。
4. Worker 解析 PDF 并生成 refined.md。
5. LLM 给出分类建议。
6. Gate 2 决定 accept / modify / reject。
7. 导出到正式库或 Obsidian。

## 7. 存储策略

1. 当前阶段采用本地数据库 + JSON 的混合存储。
2. 本地数据库保存核心状态、索引和查询字段。
3. JSON 保存阶段性结构化结果、导出元信息和可回放配置。
4. 文件系统存 PDF、Markdown、图片。
5. 后续再把 JSON 逐步收敛到正式数据库表结构。
6. 所有路径必须在 workspace 内。
7. 迁移前检查目标冲突。
8. 用户手写内容不允许被静默覆盖。
